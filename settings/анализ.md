# Анализ документа "Стандарты кода"

## 1. Все пункты документа

### I. Правила именования
1. Аббревиатуры в CamelCase (`HtmlColumn`, не `HTMLColumn`)
2. Константы в `UPPER_CASE_WITH_UNDERSCORES`
3. Элементы enum в PascalCase
4. Имена переменных/параметров отражают смысловую нагрузку (включая суффиксы `Id`, `Name`, `Flag`)
5. Именование Map в формате `keyToValueMap`
6. Имя дочернего класса сохраняет суффикс предка
7. Обобщенный тип как суффикс переменной (`conditionList`, не `listOfConditions`)

### II. Организация java-файла
8. Порядок: константы -> поля -> абстрактные методы -> обычные методы -> статические методы
9. Единый порядок параметров в связанных методах; соответствие порядка полей и параметров конструктора

### III. Документация/комментарии
10. TODO запрещены — вместо них задачи в Jira
11. Комментарии должны быть понятны всем
12. Закомментированный код запрещен
13. Deprecated-код должен сопровождаться задачей на удаление

### IV. Логирование
14. Использовать фреймворк логирования (не `System.out.println`)
15. Каждый класс — свой логгер
16. Правила логирования (кратко, информативно, формат параметров)
17. Когда добавлять логирование (else-ветки, нелогируемые фреймворком действия)
18. Уровни логирования: ERROR / WARN / INFO / DEBUG

### V. Обработка исключений
19. Либо логировать, либо пробрасывать — не оба
20. Нельзя игнорировать исключения (пустой catch)
21. В finally не должно быть исключений
22. Ловить конкретные типы исключений, не `Exception`/`Throwable`
23. Логирование исключения с полным stacktrace и контекстом
24. При пробрасывании — оборачивать в более информативное исключение
25. Обработка доп. кода в catch с защитой от потери исходного исключения
26. Когда выбрасывать исключение (ошибка пользователя или действия разработчика/админа)
27. Выбор типа исключения (стандартные или собственные с документацией)
28. Не использовать checked исключения в новом коде

### VI. Объявление и инициализация полей/переменных
29. Поля инициализировать при объявлении (в конструкторе — только при сложной логике или внешних данных)
30. Константы нельзя инициализировать через методы (за исключениями)
31. Внедрение зависимостей через конструктор (поля — `final`)
32. Нельзя объявлять переменную без значения
33. Нельзя переприсваивать значение переменной
34. Всегда указывать параметры типов (generics)
35. Использовать `final` для полей, инициализируемых однажды
36. Нельзя хранить временные данные в полях синглтона

### VII. Область видимости
37. Минимально возможная область видимости методов и полей

### VIII. Параметры методов
38. Не использовать `boolean` в параметрах (кроме сеттеров)
39. Не передавать `null` в параметрах
40. Не передавать `Optional` в параметрах
41. Правильный выбор типа коллекции: `Set` / `List` / `Iterable`
42. Вложенные вызовы методов в параметрах — выносить в переменные

### IX. Результат метода
43. Использовать ранний возврат (multiple return)
44. Метод не должен возвращать `null` — использовать `Optional` или пустую коллекцию
45. Не возвращать магические строки/числа
46. Не игнорировать результат родительского метода
47. Возвращать иммутабельные коллекции или копии
48. Кэши должны возвращать `unmodifiable` коллекции

### X. Статические поля и методы
49. Не создавать статические методы в новом коде (с исключениями: factory methods, Helper/Utility, enum)
50. Нельзя внедрять зависимости в статическое поле

### XI. Избыточный код
51. Не должно быть избыточных переменных
52. Не должно быть дублирования кода
53. Не дублировать enum с ComboValue

### XII. Нюансы Java Core
54. Не использовать массивы — использовать коллекции
55. Указание типов коллекций: стандартная реализация — интерфейс; нестандартная — конкретный тип
56. Ключи `HashMap` — классы с корректными `equals`/`hashCode` (иммутабельными)
57. Не получать коллекции для итерации в `for` — использовать `forEach`
58. Не использовать double-brace инициализацию Map
59. `Set` для хранения идентификаторов (не `List`)
60. Проверять `CollectionUtils` перед использованием Stream API

### XIII. Optional и Stream
61. Избегать `Optional#get` — использовать `ifPresent`/`isPresent`
62. Использовать `orElseGet` вместо `orElse`
63. Не использовать `peek`/`map` для побочных эффектов с non-exhaustive терминальными методами
64. Осторожно с `retainAll`, `addAll`, `removeAll`, `containsAll`

### XIV. Строки и значения
65. Идентификаторы — только латиница, цифры, подчеркивание
66. Магические строки/числа -> константы
67. Пустую строку `""` указывать напрямую (не через константу)

### XV. Прочее
68. Сравнение enum через `==`, не `equals`
69. Deprecated методы запрещены в новом коде
70. Использовать существующие методы для проверок (`isEmpty()` вместо `size() > 0`)
71. Нельзя создавать пустые не абстрактные методы для переопределения
72. Нельзя переопределять метод пустой реализацией
73. Сложные логические выражения — выносить в переменную/метод
74. Переопределенный метод должен использовать свои параметры
75. Запрет перекрываемых методов, возвращающих итераторы для обновления
76. Избегать явного приведения типов
77. Создавать иммутабельные классы
78. Конструирование объектов через конструктор (не через сеттеры)
79. Нельзя вводить prod-код специально для тестов
80. Нельзя менять область видимости ради тестов
81. Очистка кэша только после изменения данных
82. Нельзя вызывать удаленную систему в рамках транзакции
83. Избегать `switch` — использовать полиморфизм/enum
84. На старте приложения — die hard вместо fail fast

## 3. Проектно-специфичные пункты (Mirapolis)

Следующие пункты привязаны к конкретному проекту и не являются универсальными:

- **6** — суффикс предка с исключением для `ComboValue`
- **10** — TODO -> задачи в Jira (конкретный инструмент)
- **13** — Deprecated + задача на удаление в конкретном трекере
- **16-17** — формат логов и использование `org.mirapolis.log.Log`
- **30** — исключения для `generateName`/`CollectionUtils` проекта
- **46** — игнорирование `XmlActionResult`
- **50** — внедрение через `ServiceFactory.getService`
- **53** — дублирование enum `ComboValue`
- **57** — использование `forEach` вместо `for` (специфичное требование проекта)
- **60** — `CollectionUtils` проекта вместо Stream API
- **64** — ограничения `retainAll`/`addAll`/`removeAll` (проектный контекст)
- **75** — итераторы `forUpdate`
- **84** — die hard на старте приложения (специфика Mirapolis)
