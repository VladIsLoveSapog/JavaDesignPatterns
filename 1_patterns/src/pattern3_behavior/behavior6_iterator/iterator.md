# Iterator

## Итератор

**Итератор** — это поведенческий паттерн проектирования, который даёт возможность последовательно обходить элементы
составных объектов, не раскрывая их внутреннего представления.

**Коллекции** — самая частая структура данных, которую вы можете встретить в программировании. Это набор объектов,
собранный в одну кучу по каким-то причинам.

#### Основная идея

Паттерн Итератор отделяет алгоритм перебора элементов коллекции от самой коллекции, позволяя использовать один и тот же
алгоритм для различных коллекций. Это упрощает работу с коллекциями и повышает их гибкость.

#### Аналогия из жизни

Представьте пульт от телевизора с кнопками «вперёд» и «назад». Вы переключаете каналы один за другим, не зная и не
заботясь о том, как именно телевизор хранит список каналов — в массиве, связном списке или получает их со спутника.
Пульт — это итератор: он предоставляет простой интерфейс для последовательного обхода, скрывая внутреннюю структуру.

Другой пример — закладка в книге. Она запоминает текущую позицию чтения и позволяет продолжить с того места, где вы
остановились. При этом книга не меняется — закладка лишь отслеживает прогресс обхода страниц.

## Применимость

- Когда у вас есть сложная структура данных, и вы хотите скрыть от клиента детали её реализации (из-за сложности или
  вопросов безопасности).
- Когда вам нужно иметь несколько вариантов обхода одной и той же структуры данных.
- Когда вам хочется иметь единый интерфейс обхода различных структур данных.
- Когда элементы генерируются лениво (на лету), и заранее неизвестно их количество.

### Реализация

1. Определите интерфейс итератора с методами `hasNext()` и `next()`. При необходимости добавьте `remove()`.
2. Определите интерфейс коллекции (агрегата) с фабричным методом `iterator()`, который возвращает итератор.
3. Реализуйте конкретный итератор, который хранит ссылку на коллекцию и текущую позицию обхода. Каждый вызов `next()`
   продвигает позицию и возвращает очередной элемент.
4. Реализуйте метод `iterator()` в конкретной коллекции — он должен создавать и возвращать экземпляр конкретного
   итератора, связанный с данной коллекцией.
5. В клиентском коде используйте итератор через интерфейс, не обращаясь к внутреннему устройству коллекции.
6. При необходимости добавьте несколько реализаций итераторов для разных способов обхода (прямой, обратный, с фильтром).

### Iterator в Java Collections Framework

В Java паттерн Итератор является фундаментальной частью Collections Framework и реализован через два ключевых интерфейса.

#### Иерархия интерфейсов

```
Iterable<T>                          — фабричный метод iterator()
  └── Collection<T>                  — базовый интерфейс коллекций
        ├── List<T>                  — упорядоченная коллекция (ArrayList, LinkedList)
        ├── Set<T>                   — коллекция уникальных элементов (HashSet, TreeSet)
        └── Queue<T>                 — очередь (PriorityQueue, ArrayDeque)
```

#### Интерфейс `Iterator<E>`

Предоставляет методы для последовательного обхода элементов:

- `hasNext()` — проверяет, есть ли ещё элементы для итерации;
- `next()` — возвращает следующий элемент;
- `remove()` — удаляет последний возвращённый элемент (необязательный метод).

#### Интерфейс `Iterable<T>`

Определяет единственный метод `iterator()`, возвращающий объект `Iterator`. Любой класс, реализующий `Iterable`,
может использоваться в цикле `for-each`.

#### `ListIterator<E>`

Расширяет `Iterator`, добавляя двунаправленный обход (`hasPrevious()`, `previous()`), а также возможность
вставки и замены элементов (`add()`, `set()`). Доступен только для `List`.

#### Fail-fast поведение

Итераторы стандартных коллекций являются *fail-fast*: если коллекция структурно изменена после создания итератора
(не через сам итератор), выбрасывается `ConcurrentModificationException`. Это позволяет быстро обнаружить ошибку
вместо непредсказуемого поведения.

#### `for-each` как синтаксический сахар

Цикл `for-each` компилируется в вызовы `iterator()`, `hasNext()` и `next()`:

```
for (String s : list) { ... }
```

эквивалентен:

```
Iterator<String> it = list.iterator();
while (it.hasNext()) { String s = it.next(); ... }
```

### Примеры

#### Примеры из стандартной библиотеки Java

- **`ArrayList`** (`java.util`) — итератор последовательно обходит элементы массива. `ListIterator` дополнительно
  поддерживает двунаправленный обход и модификацию.
- **`HashSet`** (`java.util`) — итератор обходит элементы множества в порядке, определяемом хеш-таблицей
  (без гарантии порядка вставки).
- **`Scanner`** (`java.util`) — реализует `Iterator<String>`, позволяя последовательно читать токены из входного потока.
- **`Stream.iterator()`** (`java.util.stream`) — преобразует поток в итератор, что полезно при интеграции Stream API
  с кодом, ожидающим `Iterator`.

#### [Пример 2](code%2Fexample2_number_range%2FMain.java) — Числовой диапазон: ленивая генерация без хранимой коллекции

Пример показывает, что паттерн Итератор применим не только к коллекциям с заранее известными элементами. Класс
`NumberRange` реализует `Iterable<Integer>`, но не хранит элементы в памяти — каждое значение вычисляется «на лету»
при вызове `next()`. Это демонстрирует ленивую генерацию данных.

| ConcreteIterator        | Анонимный класс в `NumberRange.iterator()`                                          | Лениво генерирует числа арифметической прогрессии           |
| Клиент                  | [Main](code%2Fexample2_number_range%2FMain.java)                                   | Демонстрация обхода через `for-each` и явный `Iterator`     |

### Плюсы данного паттерна

- **Упрощение классов хранения данных:** Логика обхода вынесена в отдельный объект-итератор.
- **Различные способы обхода:** Позволяет реализовать несколько итераторов для одной коллекции (прямой, обратный,
  с фильтром).
- **Одновременный обход:** Позволяет одновременно перемещаться по структуре данных в разные стороны, так как каждый
  итератор хранит собственное состояние.
- **Единый интерфейс:** Клиентский код работает с любыми коллекциями через `Iterator`, не зная их внутреннего устройства.
- **Ленивая генерация:** Итератор может вычислять элементы по требованию, не храня их в памяти (как в примере 2).

### Недостатки данного паттерна

- **Дополнительные накладные расходы:** Создание и использование итераторов требует дополнительной памяти и времени.
- **Необходимость синхронизации при одновременном доступе:** В многопоточных приложениях может потребоваться
  синхронизация доступа к коллекциям при использовании итераторов.
- **Ограниченная функциональность:** Итераторы обычно предоставляют только последовательный доступ к элементам, что
  может быть недостаточно для некоторых задач (например, произвольный доступ по индексу).
- **Необходимость правильного использования методов `hasNext()` и `next()`:** Ошибки в логике перебора могут
  привести к пропуску элементов или выбросу исключений.

### Заключение

Паттерн Итератор (Iterator) является фундаментальным поведенческим паттерном, который предоставляет стандартный способ
перебора элементов коллекций без раскрытия их внутренней структуры. Он способствует снижению связанности между
клиентским кодом и коллекциями, обеспечивая гибкость и расширяемость системы.

Как показывают два реализованных примера, паттерн применим в различных сценариях: от обхода хранимой коллекции
(пример 1 — собственный ArrayList) до ленивой генерации элементов без выделения памяти под хранение (пример 2 —
числовой диапазон). В Java этот паттерн глубоко интегрирован в Collections Framework через интерфейсы `Iterable`
и `Iterator`, а цикл `for-each` делает его использование максимально удобным.
