# Memento

## Снимок

**Снимок** — это поведенческий паттерн проектирования, который позволяет делать снимки состояния объектов, не раскрывая
подробностей их реализации. Затем снимки можно использовать, чтобы восстановить прошлое состояние объектов.

#### Основная идея

Паттерн Снимок разделяет ответственность между тремя участниками:

- **Создатель (Originator)** — объект, чьё состояние необходимо сохранять. Он умеет создавать снимки и
  восстанавливаться из них.
- **Снимок (Memento)** — неизменяемый объект, хранящий копию внутреннего состояния Создателя. Снимок не предоставляет
  внешнему коду доступ к данным — только Создатель может их прочитать.
- **Опекун (Caretaker)** — объект, который хранит историю снимков и решает, когда сохранять и когда восстанавливать
  состояние. Опекун не имеет доступа к содержимому снимков.

Такое разделение позволяет сохранять и восстанавливать состояние объекта без нарушения его инкапсуляции.

#### Аналогия из жизни

Нажатие **Ctrl+Z** в текстовом редакторе. Каждый раз, когда вы вводите текст, редактор сохраняет «снимок» документа.
Нажимая Ctrl+Z, вы просите редактор вернуться к предыдущему снимку. Сам снимок — это просто копия текста, а редактор
(Опекун) хранит стек таких копий, не зная деталей их содержимого.

Другой пример — **сохранения в видеоиграх**. Игрок может сохранить прогресс (позицию, очки, инвентарь) в любой момент,
а затем загрузить сохранение, если что-то пошло не так. Система сохранений (Опекун) хранит файлы сохранений (Снимки),
а сама игра (Создатель) знает, как создать снимок своего состояния и восстановиться из него.

#### Применение

Паттерн Снимок рекомендуется использовать в следующих случаях:

- **Необходимо сохранить предыдущее состояние объекта:** Когда требуется иметь возможность вернуть объект к предыдущему
  состоянию (например, в редакторах с функцией отмены).
- **Хранение истории изменений:** Для реализации журналов или историй изменений.
- **Избегание нарушения инкапсуляции:** Когда необходимо сохранять состояние объекта без раскрытия его внутренней
  структуры.

### Реализация

1. Определите класс создателя, объекты которого должны создавать снимки своего состояния.
2. Создайте класс снимка и опишите в нём все те же поля, которые имеются в оригинальном классе-создателе.
3. Сделайте объекты снимков неизменяемыми. Они должны получать начальные значения только один раз, через свой
   конструктор.
4. Если ваш язык программирования это позволяет, сделайте класс снимка вложенным в класс создателя.
5. Добавьте в класс создателя метод получения снимков. Создатель должен создавать новые объекты снимков, передавая
   значения своих полей через конструктор.
6. Добавьте в класс создателя метод восстановления из снимка.
7. Опекуны, будь то история операций, объекты команд или нечто иное, должны знать о том, когда запрашивать снимки у
   создателя, где их хранить, и когда восстанавливать.
8. Связь опекунов с создателями можно перенести внутрь снимков. В этом случае каждый снимок будет привязан к своему
   создателю и должен будет сам восстанавливать его состояние. Но это будет работать либо если классы снимков вложены в
   классы создателей, либо если создатели имеют сеттеры для установки значений своих полей.

### Примеры

#### Примеры из стандартной библиотеки Java

- **`javax.swing.undo.UndoManager`** — менеджер отмены/повтора в Swing. Хранит стек объектов `UndoableEdit`
  (по сути — снимков изменений) и позволяет выполнять `undo()` / `redo()`. Классический пример Опекуна.
- **`java.io.Serializable`** — сериализация объекта в поток байтов фактически создаёт «снимок» его состояния,
  который можно сохранить на диск или передать по сети, а затем восстановить через десериализацию.
- **`java.util.Date.clone()`** — метод `clone()` создаёт копию объекта `Date`, что концептуально эквивалентно
  созданию снимка его состояния.

Простейшая реализация паттерна. Состояние — одна строка (`String`). Опекун хранит стек снимков

| Роль в паттерне      | Класс                                                                      | Описание                                              |

| Создатель (Originator) | [TextWindow](code%2Fexample1_text_editor%2FTextWindow.java)             | Текстовое окно — создаёт и восстанавливает снимки     |
| Опекун (Caretaker)   | [TextEditor](code%2Fexample1_text_editor%2FTextEditor.java)               | Текстовый редактор — хранит историю снимков           |
| Клиент               | [Main](code%2Fexample1_text_editor%2FMain.java)                           | Демонстрация ввода, сохранения и отмены               |

#### [Пример 2](code%2Fexample2_game_state%2FMain.java) — Состояние игры: несколько полей

Состояние содержит несколько примитивных полей (позиция игрока, счёт). Опекун поддерживает
многоуровневый **undo** — можно откатиться на несколько сохранений назад.

| Роль в паттерне      | Класс                                                                      | Описание                                               |
|----------------------|----------------------------------------------------------------------------|--------------------------------------------------------|

| Клиент               | [Main](code%2Fexample2_game_state%2FMain.java)                            | Демонстрация сохранения и восстановления игры          |

#### [Пример 3](code%2Fexample3_drawing_editor%2FMain.java) — Графический редактор: Undo + Redo

Наиболее полная реализация. Состояние — коллекция объектов (`List<Shape>`). Используются два стека
(undo + redo) для полноценной навигации по истории. При новом действии стек redo очищается.

| Роль в паттерне      | Класс                                                                            | Описание                                              |
|----------------------|----------------------------------------------------------------------------------|-------------------------------------------------------|
| Объект-значение      | [Shape](code%2Fexample3_drawing_editor%2FShape.java)                            | Неизменяемая фигура (тип, координаты, цвет)           |
| Снимок (Memento)     | [CanvasMemento](code%2Fexample3_drawing_editor%2FCanvasMemento.java)            | Защитная копия списка фигур                           |
| Создатель (Originator) | [Canvas](code%2Fexample3_drawing_editor%2FCanvas.java)                        | Холст — создаёт и восстанавливает снимки              |
| Опекун (Caretaker)   | [DrawingHistory](code%2Fexample3_drawing_editor%2FDrawingHistory.java)          | История рисования — undo/redo через два стека         |
| Клиент               | [Main](code%2Fexample3_drawing_editor%2FMain.java)                              | Демонстрация undo, redo и очистки redo                |

### Сравнение трёх примеров

| Критерий                  | Пример 1 (Текстовый редактор) | Пример 2 (Игра)             | Пример 3 (Графический редактор) |
|---------------------------|-------------------------------|-----------------------------|---------------------------------|
| **Тип состояния**         | Одна строка (`String`)        | Примитивы (`int`, `int`)    | Коллекция (`List<Shape>`)       |
| **Сложность состояния**   | Простое                       | Среднее                     | Сложное (составное)             |
| **Undo**                  | Да                            | Да (многоуровневый)         | Да                              |
| **Redo**                  | Нет                           | Нет                         | Да                              |
| **Тип копирования**       | `toString()` (копия строки)   | Копирование примитивов      | Защитная копия списка           |
| **Структура Опекуна**     | Один стек (`Deque`)           | Один стек (`Deque`)         | Два стека (undo + redo)         |

### Сравнение с паттерном Command

Паттерны **Снимок** и **Команда** часто используются совместно для реализации undo/redo,
но решают разные задачи:

| Критерий                | Memento (Снимок)                                 | Command (Команда)                               |
|-------------------------|--------------------------------------------------|-------------------------------------------------|
| **Что сохраняется**     | Полное состояние объекта                         | Действие (операция + параметры)                 |
| **Как отменяется**      | Восстановление из снимка                         | Вызов обратной операции (`undo()`)              |
| **Потребление памяти**  | Высокое (полная копия состояния)                 | Низкое (только описание действия)               |
| **Сложность отмены**    | Простая (замена состояния)                        | Может быть сложной (нужна обратная операция)    |
| **Когда использовать**  | Состояние сложно описать набором обратных команд | Действия легко обратимы                         |
| **Совместное использование** | Часто используется вместе с Command          | Может хранить Memento для отката                |

### Плюсы данного паттерна

- **Легкость реализации механизмов отмены и повторения:** Обеспечивает простой способ отката к предыдущим состояниям.
- **Гибкость управления историей состояний:** Можно реализовать сложные механизмы истории, такие как множественные
  уровни отмены или линейное и ветвящееся восстановление.
- **Повторное использование объектов Memento:** Один и тот же объект снимка может использоваться разными объектами
  (например перенос сохранений с одной машины на другую).

### Недостатки данного паттерна

- **Увеличение использования памяти:** Хранение многочисленных состояний может потребовать значительного объема памяти.
- **Сложность реализации:** Требует дополнительного проектирования и создания дополнительных классов (Memento).
- **Управление жизненным циклом состояний:** Необходимо контролировать сохранение и удаление состояний, чтобы избежать
  утечек памяти.

### Заключение

Паттерн Снимок (Memento) является мощным инструментом для управления состоянием объектов, особенно когда требуется
реализовать механизмы отмены и восстановления. Он позволяет сохранять состояние объекта без раскрытия его внутренней
структуры, обеспечивая при этом гибкость и расширяемость системы.

Как показывают три реализованных примера, паттерн может быть адаптирован под различные сценарии: от простого undo
со строковым состоянием (пример 1), через многоуровневую отмену с несколькими полями (пример 2), до полноценного
undo/redo со сложным составным состоянием и двумя стеками (пример 3).

При использовании паттерна Снимок важно учитывать потенциальные недостатки, такие как увеличение использования
памяти и сложность реализации. Правильное управление жизненным циклом состояний и минимизация объёма сохраняемых данных
помогут эффективно использовать этот паттерн в ваших проектах.
