# State

## Состояние

**Состояние** — это поведенческий паттерн проектирования, который позволяет объектам менять поведение в зависимости от
своего состояния. Извне создаётся впечатление, что изменился класс объекта.

Паттерн схож с конечным автоматом и предназначен для избавления от множества `if` и `switch` операторов.
Проблема `if` и `switch` операторов заключается в их негибкости и нагромождённости. Зачастую множество `if` операторов
лежит в одном месте, что ещё больше усложняет понимание контекста. При паттерне **Состояние** особенности поведения
переносят в класс состояния, тем самым убирая множественное ветвление.

*Задумайтесь об использовании **Состояния** там, где у вас много `if` и `switch` операторов*.

#### Основная идея

Паттерн Состояние разделяет функциональность объекта на отдельные классы, каждый из которых представляет определённое
состояние. Контекст (объект, поведение которого изменяется) содержит ссылку на текущий объект-состояние и делегирует ему
выполнение соответствующих действий.

#### Аналогия из жизни

Представьте светофор на перекрёстке. Он может находиться в одном из трёх состояний: красный, жёлтый, зелёный. В каждом
состоянии светофор «ведёт себя» по-разному: на красный — движение запрещено, на зелёный — разрешено, на жёлтый —
предупреждение. При этом сам светофор (контекст) не содержит логики «если красный — стой, если зелёный — иди». Вместо
этого каждое состояние знает, что делать и на какой сигнал переключиться следующим. Светофор лишь делегирует работу
текущему состоянию и хранит ссылку на него.

Другой пример — телефон. В заблокированном состоянии нажатие кнопки показывает экран блокировки, в разблокированном —
открывает приложение. Один и тот же жест приводит к разному поведению в зависимости от состояния устройства.

#### Применение

Паттерн Состояние рекомендуется использовать в следующих случаях:

- **Объект должен изменять своё поведение в зависимости от своего состояния:** Когда поведение объекта должно изменяться
  динамически при изменении его внутреннего состояния.
- **Сложные условные операторы:** Когда в классе много условных операторов (`if`, `switch`), которые изменяют
  поведение в зависимости от состояния. Паттерн Состояние помогает избежать этого, разделяя поведение на отдельные
  классы.
- **Необходимо легко добавлять новые состояния:** Паттерн облегчает добавление новых состояний без изменения
  существующего кода контекста или других состояний.

### Реализация

1. Определитесь с классом, который будет играть роль контекста. Это может быть как существующий класс, в котором уже
   есть зависимость от состояния, так и новый класс, если код состояний размазан по нескольким классам.
2. Создайте интерфейс состояний. Он должен описывать методы, общие для всех состояний, обнаруженных в контексте.
   Заметьте, что не всё поведение контекста нужно переносить в состояние, а только то, которое зависит от состояний.
3. Для каждого фактического состояния создайте класс, реализующий интерфейс состояния. Переместите весь код, связанный
   с конкретным состоянием, в нужный класс. В конце концов, все методы интерфейса состояния должны быть реализованы.
4. Создайте в контексте поле для хранения объектов-состояний, а также публичный метод для изменения значения этого поля.
5. Старые методы контекста, в которых находился зависимый от состояния код, замените на вызовы соответствующих методов
   объекта-состояния.
6. В зависимости от бизнес-логики, разместите код, который переключает состояние контекста, либо внутри контекста, либо
   внутри классов конкретных состояний.

### Примеры

#### Примеры из стандартной библиотеки Java

- **`Thread.State`** (`java.lang`) — перечисление состояний потока: `NEW`, `RUNNABLE`, `BLOCKED`, `WAITING`,
  `TIMED_WAITING`, `TERMINATED`. Хотя это `enum`, концепция управления поведением потока в зависимости от его состояния
  напоминает паттерн Состояние.
- **`JButton`** (`javax.swing`) — управляет визуальным состоянием (нормальное, нажатое, отключённое) и поведением
  в зависимости от этого состояния. Концепция изменения поведения и внешнего вида компонента в зависимости от состояния
  соответствует идее паттерна.
- **`javax.faces.lifecycle.Lifecycle`** (JSF) — жизненный цикл JSF-запроса проходит через фазы (`RESTORE_VIEW`,
  `APPLY_REQUEST_VALUES`, `PROCESS_VALIDATIONS` и др.), каждая из которых определяет своё поведение обработки.

#### [Пример 1](code%2Fexample1_order_state%2FMain.java) — Доставка посылки: линейная смена состояний

Простой пример с посылкой, состояние которой меняется линейно: **Заказана → Доставлена → Получена**. Каждое состояние
знает своего предшественника и преемника. Переход за границы (назад из начального или вперёд из конечного) является
no-op — операция игнорируется с выводом сообщения.

| Роль в паттерне       | Класс                                                                  | Описание                                                |
|-----------------------|------------------------------------------------------------------------|---------------------------------------------------------|
| Интерфейс состояния   | [PackageState](code%2Fexample1_order_state%2FPackageState.java)       | Определяет методы `next()`, `prev()`, `printStatus()`   |
| Конкретное состояние   | [OrderedState](code%2Fexample1_order_state%2FOrderedState.java)       | Посылка заказана (начальное состояние)                   |
| Конкретное состояние   | [DeliveredState](code%2Fexample1_order_state%2FDeliveredState.java)   | Посылка доставлена (промежуточное состояние)             |
| Конкретное состояние   | [ReceivedState](code%2Fexample1_order_state%2FReceivedState.java)     | Посылка получена (конечное состояние)                    |
| Контекст              | [Package](code%2Fexample1_order_state%2FPackage.java)                 | Посылка — делегирует переходы текущему состоянию         |
| Клиент                | [Main](code%2Fexample1_order_state%2FMain.java)                       | Демонстрация последовательной смены состояний            |

#### [Пример 2](code%2Fexample2_game%2FMain.java) — Игровой персонаж: нелинейные переходы

Игровой персонаж может находиться в трёх состояниях (`Standing`, `Running`, `Jumping`) и реагировать на
команды (`pressRun()`, `pressJump()`, `pressStop()`). В отличие от примера 1, переходы здесь нелинейные —
не все действия разрешены в каждом состоянии. Например, из прыжка нельзя начать бег, а остановиться стоя — бессмысленно.

| Роль в паттерне       | Класс                                                              | Описание                                                    |
|-----------------------|--------------------------------------------------------------------|-------------------------------------------------------------|
| Интерфейс состояния   | [State](code%2Fexample2_game%2FState.java)                        | Определяет методы `pressRun()`, `pressJump()`, `pressStop()`|
| Конкретное состояние   | [StandingState](code%2Fexample2_game%2FStandingState.java)        | Персонаж стоит: может бежать или прыгнуть                   |
| Конкретное состояние   | [RunningState](code%2Fexample2_game%2FRunningState.java)          | Персонаж бежит: может прыгнуть или остановиться             |
| Конкретное состояние   | [JumpingState](code%2Fexample2_game%2FJumpingState.java)          | Персонаж прыгает: может только приземлиться                 |
| Контекст              | [Player](code%2Fexample2_game%2FPlayer.java)                      | Игрок — делегирует действия текущему состоянию              |
| Клиент                | [Main](code%2Fexample2_game%2FMain.java)                          | Демонстрация нелинейных переходов между состояниями         |

#### [Пример 3](code%2Fexample3_traffic_light%2FMain.java) — Светофор: циклические состояния

Светофор циклически переключается между тремя состояниями: **Красный → Зелёный → Жёлтый → Красный → ...**
Каждое состояние знает только одного преемника, а цикл замыкается за счёт того, что жёлтый переходит
обратно в красный. Это самый простой вариант паттерна Состояние — у каждого состояния ровно один переход.

| Роль в паттерне       | Класс                                                                              | Описание                                                |
|-----------------------|------------------------------------------------------------------------------------|---------------------------------------------------------|
| Интерфейс состояния   | [TrafficLightState](code%2Fexample3_traffic_light%2FTrafficLightState.java)       | Определяет методы `switchLight()`, `getColor()`         |
| Конкретное состояние   | [RedLight](code%2Fexample3_traffic_light%2FRedLight.java)                         | Красный свет → переход на зелёный                       |
| Конкретное состояние   | [GreenLight](code%2Fexample3_traffic_light%2FGreenLight.java)                     | Зелёный свет → переход на жёлтый                        |
| Конкретное состояние   | [YellowLight](code%2Fexample3_traffic_light%2FYellowLight.java)                   | Жёлтый свет → переход на красный                        |
| Контекст              | [TrafficLight](code%2Fexample3_traffic_light%2FTrafficLight.java)                 | Светофор — делегирует переключение текущему состоянию    |
| Клиент                | [Main](code%2Fexample3_traffic_light%2FMain.java)                                 | Демонстрация полного цикла переключений                 |

### Плюсы данного паттерна

- **Упрощение кода:** Позволяет избавиться от сложных условных операторов (`if`, `switch`) и заменяет их
  полиморфизмом.
- **Гибкость:** Легко добавлять новые состояния без изменения существующего кода контекста или других состояний.
- **Снижение связанности:** Контекст не знает деталей реализации состояний, что облегчает поддержку и расширение
  системы.
- **Легкость изменения состояний во время выполнения:** Состояние объекта может изменяться динамически в зависимости от
  событий или условий.

### Недостатки данного паттерна

- **Увеличение числа классов:** Каждый новый тип состояния требует создания отдельного класса, что может привести к
  большому количеству классов.
- **Сложность отладки:** Из-за разделения логики на разные классы может быть сложнее отследить поток выполнения
  программы.
- **Необходимость управления состояниями:** Нужно обеспечить правильное управление переходами между состояниями, что
  может усложнить дизайн системы.

### Сравнение с паттерном Стратегия

Паттерны **Состояние** и **Стратегия** похожи, а их UML-диаграммы по сути одинаковы.
Однако между ними есть принципиальные различия:

| Критерий | State | Strategy |
|---|---|---|
| **Цель** | Изменить поведение при смене состояния | Выбрать один алгоритм из семейства |
| **Кто инициирует смену** | Сам объект (или состояние) переключает состояние | Клиент (внешний код) передаёт нужную стратегию |
| **Осведомлённость** | Состояния часто знают о соседних состояниях и инициируют переходы | Стратегии не знают друг о друге |
| **Количество активных** | Ровно одно состояние, но переходы — часть логики | Ровно одна стратегия в каждый момент |
| **Типичный сценарий** | Заказ (новый → оплачен → доставлен), TCP-соединение | Сортировка, маршрутизация, расчёт скидок |

Паттерн **Состояние** используется для инкапсуляции характеристик состояния. Поведение объекта зависит от его состояния —
когда объект находится в определённом состоянии, он ведёт себя иначе, чем когда находится в другом.

Паттерн **Стратегия** инкапсулирует алгоритмы. Из множества стратегий выбирается та, которая решает задачу наиболее
эффективно.

### Заключение

Паттерн Состояние (State) предоставляет мощный способ управления поведением объектов в зависимости от их внутреннего
состояния. Он улучшает гибкость и расширяемость системы, позволяя легко добавлять новые состояния и изменять поведение
без изменения основного кода контекста.

Как показывают три реализованных примера, паттерн может быть адаптирован под различные сценарии: от линейной цепочки
состояний (пример 1), через нелинейные переходы с ограничениями (пример 2), до циклических переключений (пример 3).

При использовании паттерна Состояние важно тщательно проектировать интерфейсы и классы состояний, чтобы обеспечить
максимальную гибкость и минимальную связанность между компонентами системы.
