# Template Method

## Шаблонный метод

**Паттерн Шаблонный Метод (Template Method)** относится к поведенческим паттернам и определяет скелет алгоритма
в базовом классе, делегируя реализацию отдельных шагов подклассам. Подклассы могут переопределять конкретные шаги
алгоритма, не изменяя его общую структуру и порядок выполнения.

#### Основная идея

Создать абстрактный класс с шаблонным методом, который определяет порядок выполнения шагов алгоритма. Некоторые из этих
шагов реализованы в абстрактном классе, а другие — объявлены как абстрактные методы, которые должны быть реализованы в
подклассах.

Шаблонный метод объявляется как `final`, чтобы подклассы не могли изменить последовательность шагов.
Это реализует **«Голливудский принцип»** (Hollywood Principle): «Не звоните нам — мы сами вам позвоним».
Базовый класс вызывает методы подклассов в нужный момент, а не наоборот.

#### Ключевые понятия

В паттерне Template Method выделяют четыре типа методов:

| Тип метода | Модификатор | Обязательно переопределять? | Пример |
|---|---|---|---|
| Шаблонный метод | `final` | Нет (нельзя!) | `generateDocument()`, `pass()`, `prepareRecipe()` |
| Абстрактный шаг | `abstract` | Да | `createBody()`, `brew()`, `writeProgram()` |
| Конкретный шаг | `private` / обычный | Нет | `boilWater()`, `printDocument()` |
| Хук | Обычный с пустым/default телом | Нет (по желанию) | `addTimestamp()`, `writeOff()`, `correctRemarks()` |

**Хук (hook)** — это метод с реализацией по умолчанию (часто пустой или возвращающей значение). Подклассы могут
переопределить хук, чтобы «встроиться» в алгоритм, но не обязаны это делать.

#### Применение

- **Повторяющиеся алгоритмы с вариациями:** Когда несколько классов используют один и тот же алгоритм, но с небольшими
  изменениями в его отдельных шагах.
- **Контроль над наследованием:** Когда нужно контролировать порядок выполнения шагов алгоритма и обеспечить
  последовательность их вызова.
- **Расширяемость алгоритмов:** Когда необходимо добавить новые шаги или изменить существующие без изменения общего
  алгоритма.
- **Сокрытие деталей реализации:** Когда требуется скрыть детали реализации некоторых шагов алгоритма от клиентов.

### Реализация

1. Изучите алгоритм и подумайте, можно ли его разбить на шаги. Прикиньте, какие шаги будут стандартными для всех
   вариаций алгоритма, а какие — изменчивыми.
2. Создайте абстрактный базовый класс. Определите в нём шаблонный метод. Этот метод должен состоять из вызовов шагов
   алгоритма. Имеет смысл сделать шаблонный метод финальным, чтобы подклассы не могли переопределить его (если ваш
   язык программирования это позволяет).
3. Добавьте в абстрактный класс методы для каждого из шагов алгоритма. Вы можете сделать эти методы абстрактными или
   добавить какую-то реализацию по умолчанию. В первом случае, все подклассы должны будут реализовать эти методы, а во
   втором — только если реализация шага в подклассе отличается от стандартной версии.
4. Подумайте о введении в алгоритм хуков. Чаще всего, хуки располагают между основными шагами алгоритма, а также до и
   после всех шагов.
5. Создайте конкретные классы, унаследовав их от абстрактного класса. Реализуйте в них все недостающие шаги и хуки.

### Примеры из стандартной библиотеки Java

- **`java.util.AbstractList`** — абстрактные методы `get(int)` и `size()` должны быть реализованы подклассами;
  метод `indexOf()` использует их как шаблонный метод для поиска элемента.
- **`java.io.InputStream`** — абстрактный метод `read()` (один байт) реализуется подклассами;
  метод `read(byte[], int, int)` вызывает `read()` в цикле — это шаблонный метод.
- **`javax.servlet.http.HttpServlet`** — метод `service()` определяет алгоритм обработки HTTP-запроса
  и делегирует конкретные методы `doGet()`, `doPost()` и т.д. подклассам (хуки с реализацией по умолчанию).

### Примеры

#### [Пример 1](code%2Fexample1_lab%2FMain.java) — Сдача лабораторной работы

Юмористический пример: студент сдаёт лабораторную работу по СДП. Алгоритм сдачи фиксирован в шаблонном методе
`pass()`, но отличается для честного и нечестного студента. Хуки `writeOff()` и `correctRemarks()` позволяют
«встроиться» в алгоритм при необходимости.

| Класс | Роль в паттерне | Описание |
|---|---|---|
| `IPassLabWork` | Интерфейс | Контракт алгоритма сдачи ЛР |
| `PassLabWork` | AbstractClass | Шаблонный метод `pass()` (`final`), хуки `writeOff()` и `correctRemarks()` |
| `HonestlyPassSDPLabWork` | ConcreteClass | Честная сдача — хуки не переопределены |
| `NoHonestlyPassSDPLabWork` | ConcreteClass | Нечестная сдача — оба хука переопределены |
| `Main` | Клиент | Демонстрирует оба варианта сдачи |

---

#### [Пример 2](code%2Fexample2_drinks%2FMain.java) — Приготовление напитков

Классический пример: приготовление чая и кофе. Общий алгоритм (вскипятить воду → заварить → налить → добавить добавки)
определён в шаблонном методе `prepareRecipe()`. Конкретные шаги `boilWater()` и `pourInCup()` одинаковы для всех
напитков, а абстрактные шаги `brew()` и `addCondiments()` реализуются подклассами.

| Класс | Роль в паттерне | Описание |
|---|---|---|
| `Beverage` | AbstractClass | Шаблонный метод `prepareRecipe()` (`final`), конкретные шаги `boilWater()`, `pourInCup()` |
| `Tea` | ConcreteClass | Заваривание чая, добавление лимона |
| `Coffee` | ConcreteClass | Заваривание кофе, добавление сахара и молока |
| `Main` | Клиент | Демонстрирует приготовление обоих напитков |

---

#### [Пример 3](code%2Fexample3_document_generator%2FMain.java) — Генерация документов

Пример с хуком: генерация документов в разных форматах (HTML, plain text). Шаблонный метод `generateDocument()`
определяет алгоритм: создать заголовок → создать тело → (опционально) добавить метку времени → напечатать.
Хук `addTimestamp()` позволяет подклассам отключить добавление метки времени.

| Класс | Роль в паттерне | Описание |
|---|---|---|
| `AbstractDocumentGenerator` | AbstractClass | Шаблонный метод `generateDocument()` (`final`), хук `addTimestamp()`, конкретные шаги `getTimestamp()`, `printDocument()` |
| `HtmlDocumentGenerator` | ConcreteClass | HTML-формат, хук не переопределён — метка времени добавляется |
| `PlainTextDocumentGenerator` | ConcreteClass | Текстовый формат, хук переопределён → метка времени не добавляется |
| `Main` | Клиент | Демонстрирует оба генератора и разницу в поведении хука |

### Сравнение примеров

| Критерий | Пример 1 (ЛР) | Пример 2 (Напитки) | Пример 3 (Документы) |
|---|---|---|---|
| Хуки | `writeOff()`, `correctRemarks()` | Нет | `addTimestamp()` |
| Абстрактные шаги | 4 | 2 | 2 |
| Конкретные шаги | Нет | `boilWater()`, `pourInCup()` | `printDocument()`, `getTimestamp()` |
| Интерфейс | Да (`IPassLabWork`) | Нет | Нет |

### Сравнение с паттерном Стратегия

| Критерий | Template Method | Strategy |
|---|---|---|
| Механизм | Наследование | Композиция |
| Кто определяет алгоритм | Базовый класс (скелет фиксирован) | Клиент передаёт стратегию |
| Изменение в runtime | Нет (класс выбирается при создании) | Да (стратегию можно заменить) |
| Гранулярность | Переопределяются отдельные шаги | Заменяется весь алгоритм целиком |
| Связь между классами | Тесная (наследование) | Слабая (композиция) |

### Плюсы данного паттерна

- **Повторное использование кода:** Общие шаги алгоритма реализованы в абстрактном классе, что позволяет избежать
  дублирования кода в подклассах.
- **Контроль над порядком выполнения шагов:** Шаблонный метод определяет последовательность выполнения шагов,
  обеспечивая правильный порядок действий.
- **Гибкость и расширяемость:** Подклассы могут переопределять только необходимые шаги алгоритма, не затрагивая его
  общую структуру.
- **Сокрытие деталей реализации:** Клиентский код взаимодействует с абстрактным классом, не зная о конкретных
  реализациях шагов алгоритма.

### Недостатки данного паттерна

- **Жесткая структура алгоритма:** Алгоритм задан в шаблонном методе и может быть сложно изменить его структуру, если
  это потребуется.
- **Увеличение числа классов:** Для каждой новой вариации алгоритма необходимо создавать новый подкласс.
- **Сложность понимания:** Разделение алгоритма на абстрактный класс и подклассы может затруднить понимание общего
  алгоритма.
- **Ограниченная гибкость при использовании композиции:** Паттерн ориентирован на наследование, что может привести к
  проблемам при необходимости использования композиции вместо наследования.

### Заключение

**Паттерн Шаблонный Метод (Template Method)** позволяет вынести общий алгоритм в базовый класс, а детали реализации
делегировать подклассам. Все три примера демонстрируют эту идею: алгоритм сдачи ЛР (`pass()`), приготовление напитков
(`prepareRecipe()`) и генерация документов (`generateDocument()`) — каждый из них фиксирует последовательность шагов
в `final`-методе и позволяет подклассам варьировать отдельные этапы.

Особую роль играют **хуки**: в примере с ЛР они позволяют «списать» и «исправить замечания», а в примере с документами —
отключить добавление метки времени. Хуки дают подклассам возможность влиять на поведение алгоритма без изменения его
структуры.

При выборе между Template Method и Strategy стоит учитывать: если нужно зафиксировать скелет алгоритма и варьировать
лишь отдельные шаги — подходит Template Method; если нужно заменять алгоритм целиком во время выполнения — лучше
использовать Strategy.
