# Decorator

## Декоратор

**Декоратор** — это структурный паттерн проектирования, который позволяет динамически добавлять объектам новую
функциональность, оборачивая их в полезные «обёртки».

Паттерн Декоратор является лучшим примером того, когда использование **композиции превосходит наследование**.

В разработке программного обеспечения часто возникает необходимость динамически добавлять объектам новые возможности
без изменения их структуры. Паттерн Декоратор предоставляет гибкое решение этой задачи, позволяя расширять поведение
объектов путём оборачивания их в дополнительные компоненты.

#### Основная идея

Основная идея паттерна Декоратор заключается в том, чтобы предоставить способ оборачивания объектов для добавления новых
функций, сохраняя при этом интерфейс исходного объекта. Это позволяет клиентам взаимодействовать с декорированными
объектами так же, как с оригинальными, не зная о дополнительном функционале.

#### Композиция vs Наследование

Почему нельзя просто создать подкласс и добавить нужное поведение?

- **Наследование статично** — нельзя изменить поведение существующего объекта в runtime. Нужно заменить весь объект
  другим, созданным из другого подкласса.
- **Комбинаторный взрыв** — если нужно комбинировать несколько расширений, количество подклассов растёт экспоненциально.
  Например, 3 независимые опции дают 2^3 = 8 подклассов.
- **Декоратор решает обе проблемы** — поведение добавляется динамически, а комбинации создаются путём вложения
  декораторов друг в друга.

#### Применение

Примеры, когда стоит использовать паттерн Декоратор:

1. Когда нужно добавить, улучшить или убрать поведение объекта **динамически, в runtime**.
2. Когда нужно менять обязанности объектов на лету, незаметно для кода, который их использует.
3. Когда нельзя расширить обязанности объекта с помощью наследования (например, класс `final`).
4. Когда нужно изменить функциональность одного конкретного объекта, а остальные оставить без изменений.

#### Декоратор в стандартной библиотеке Java

Самый известный пример паттерна Декоратор в Java — это **иерархия потоков ввода-вывода (java.io)**:

```
new BufferedReader(new InputStreamReader(new FileInputStream("file.txt")))
```

Здесь `FileInputStream` — конкретный компонент, `InputStreamReader` — декоратор, добавляющий перекодировку символов,
а `BufferedReader` — декоратор, добавляющий буферизацию. Каждый «слой» добавляет функциональность, не меняя предыдущий.

### Структура паттерна

Паттерн Декоратор состоит из четырёх участников:

| Участник | Роль | Пример из example1 |
|----------|------|--------------------|
| **Component** (интерфейс) | Общий интерфейс для компонентов и декораторов | `Pizza` |
| **Concrete Component** | Базовый объект с основной логикой | `PepperoniPizza` |
| **Base Decorator** | Абстрактный класс, хранящий ссылку на обёрнутый объект | `PizzaDecorator` |
| **Concrete Decorator** | Добавляет конкретную функциональность | `CheesePizzaDecorator` |

### Реализация

1. Убедитесь, что в вашей задаче есть один основной компонент и несколько опциональных дополнений или надстроек над ним.
2. Создайте интерфейс компонента, который описывает все общие методы как для основного компонента, так и для его
   дополнений.
3. Создайте класс конкретного компонента и поместите в него основную бизнес-логику.
4. Создайте базовый класс декораторов. Он должен иметь поле для хранения ссылки на вложенный объект-компонент. Все
   методы базового декоратора должны делегировать действие вложенному объекту.
5. И конкретный компонент, и базовый декоратор должны следовать одному и тому же интерфейсу компонента.
6. Создайте классы конкретных декораторов, наследуя их от базового декоратора. Конкретный декоратор должен
   выполнять свою добавочную функциональность, а затем (или перед этим) вызывать эту же операцию обёрнутого объекта.
7. Клиент берёт на себя ответственность за конфигурацию и порядок обёртывания объектов.

### Примеры

#### [Пример 1](code%2Fexample1_pizza%2FMain.java) — Пиццерия (добавление и удаление ингредиентов)

Демонстрирует базовый принцип паттерна на понятном примере — конфигурация пиццы.

- `Pizza` — интерфейс компонента с методами `getDescription()` и `getCost()`.
- `PepperoniPizza` — конкретный компонент (базовая пицца).
- `PizzaDecorator` — базовый декоратор, хранящий ссылку на обёрнутую пиццу.
- `CheesePizzaDecorator` — декоратор, добавляющий сыр (расширяет описание и цену).
- `NonCheeseDecorator` — декоратор, убирающий сыр. Показывает, что декоратор может не только **добавлять**,
  но и **убирать** функциональность.

Пример демонстрирует вложение декораторов: пиццу сначала оборачивают `CheesePizzaDecorator`, а затем
`NonCheeseDecorator` — и мы возвращаемся к исходному состоянию.

#### [Пример 2](code%2Fexample2_message%2FMessageMain.java) — Обработка сообщений (шифрование и временные метки)

Показывает, как декораторы могут трансформировать содержимое объекта.

- `Message` — интерфейс компонента с методом `getContent()`.
- `SimpleMessage` — конкретный компонент (простое текстовое сообщение).
- `MessageDecorator` — базовый декоратор, делегирующий вызов обёрнутому сообщению.
- `EncryptedMessageDecorator` — декоратор, «шифрующий» текст (симуляция — разворот строки).
- `TimestampedMessageDecorator` — декоратор, добавляющий временную метку к тексту.

Важный момент: **порядок оборачивания влияет на результат**. Если сначала зашифровать, а потом добавить метку —
метка будет читаемой. Если наоборот — метка тоже зашифруется.

#### [Пример 3](code%2Fexample3_data_stream%2FDataStreamMain.java) — Потоки данных (сжатие и Base64-кодирование)

Пример, приближённый к реальному использованию паттерна в java.io.

- `DataStream` — интерфейс компонента с методами `write()` и `read()`.
- `StringDataStream` — конкретный компонент (хранит данные в строке).
- `DataStreamDecorator` — базовый декоратор, аналог `FilterInputStream` из java.io.
- `CompressionDecorator` — декоратор, имитирующий сжатие данных при записи.
- `Base64EncodingDecorator` — декоратор, кодирующий данные в Base64 при записи и декодирующий при чтении.

Этот пример показывает:
- **Симметричную обработку** — декоратор трансформирует данные и при записи, и при чтении.
- **Комбинирование декораторов** — сжатие + Base64 работают вместе, каждый добавляя свой слой обработки.
- **Аналогию с java.io** — конструкция `new CompressionDecorator(new Base64EncodingDecorator(new StringDataStream()))`
  аналогична `new GZIPOutputStream(new BufferedOutputStream(new FileOutputStream("file.gz")))`.

### Плюсы паттерна

- **Гибкость:** позволяет динамически добавлять функциональность объектам без изменения их структуры.
- **Повторное использование:** можно комбинировать различные декораторы для создания нужного поведения.
- **Принцип единственной ответственности:** каждый декоратор отвечает за одну конкретную функцию.
- **Принцип открытости/закрытости:** новые декораторы можно добавлять, не меняя существующий код.
- **Прозрачность для клиента:** клиент работает с объектом через общий интерфейс и не знает, сколько
  «слоёв» декораторов обёрнуто вокруг него.

### Минусы паттерна

- **Увеличение числа классов:** каждый новый декоратор — это отдельный класс.
- **Сложность отладки:** множественные обёртки усложняют трассировку вызовов в отладчике.
- **Порядок имеет значение:** неправильный порядок оборачивания может привести к неожиданным результатам
  (как показано в примере 2 с шифрованием и метками).
- **Трудно снять конкретный декоратор:** если объект обёрнут в несколько слоёв, нельзя просто убрать
  один из средних — нужно пересобрать всю цепочку.

### Декоратор vs другие паттерны

| Сравнение | Декоратор | Другой паттерн |
|-----------|-----------|----------------|
| vs **Адаптер** | Не меняет интерфейс, добавляет поведение | Адаптер преобразует один интерфейс в другой |
| vs **Прокси** | Добавляет функциональность | Прокси контролирует доступ к объекту |
| vs **Стратегия** | Меняет «внешность» объекта (обёртка) | Стратегия меняет «внутренности» (подстановка алгоритма) |
| vs **Наследование** | Динамическое расширение в runtime | Статическое расширение на этапе компиляции |

### Заключение

Паттерн Декоратор — мощный инструмент для динамического расширения функциональности объектов без
изменения их структуры. Он широко применяется в стандартной библиотеке Java (java.io, Collections),
обеспечивая гибкость и модульность.

Правильное применение паттерна способствует созданию гибких и расширяемых систем, облегчая добавление
новых возможностей. Однако необходимо учитывать увеличение числа классов и сложность отладки,
чтобы эффективно использовать этот паттерн.
