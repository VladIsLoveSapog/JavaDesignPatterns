# Adapter

## Адаптер

**Адаптер** — это структурный паттерн проектирования, который позволяет объектам с несовместимыми интерфейсами работать
вместе.

В разработке программного обеспечения часто возникает необходимость интегрировать различные компоненты и системы,
которые изначально не предназначены для совместной работы. Паттерн проектирования **Адаптер** (Adapter) предоставляет
способ сделать это, преобразуя интерфейс одного класса в интерфейс, ожидаемый клиентом. Это позволяет классам с
несовместимыми интерфейсами работать вместе без изменения их исходного кода.

#### Аналогия из жизни

Простейший пример адаптера — переходник для электрической розетки. Вы привезли ноутбук с европейской вилкой (два
круглых штырька), а в гостинице розетки британского стандарта (три плоских контакта). Вилку и розетку вы изменить
не можете — они «чужие». Но маленький переходник (адаптер) позволяет им работать вместе, преобразуя один физический
интерфейс в другой.

В программировании всё то же самое: есть полезный класс (розетка), есть наш код (вилка), и их интерфейсы не совпадают.
Адаптер — это тот самый переходник.

#### Основная идея

Основная идея паттерна Адаптер заключается в том, чтобы предоставить промежуточный слой между двумя несовместимыми
интерфейсами, позволяющий им взаимодействовать без изменения их исходного кода.

#### Применение

Паттерн Адаптер рекомендуется использовать в следующих случаях:

- Необходимо использовать существующий класс, но его интерфейс несовместим с текущей системой.
- Разработка новых классов, совместимых с существующей системой.
- Интеграция сторонних библиотек или систем, имеющих другие интерфейсы.
- Превращение интерфейса одного класса в другой, ожидаемый клиентом.

### Участники паттерна

| Участник | Роль | В примере с сортировкой |
|----------|------|------------------------|
| **Target** (целевой интерфейс) | Интерфейс, с которым работает клиентский код | `Sorter` |
| **Adaptee** (адаптируемый класс) | Существующий класс с полезным функционалом, но несовместимым интерфейсом | `SorterExternalProduct` |
| **Adapter** (адаптер) | Класс-обёртка, реализующий Target и делегирующий работу Adaptee | `SorterAdapter` |
| **Client** (клиент) | Код, который работает с Target и не знает об Adaptee | `SortAdapterMain` |

### Реализация

1. Убедитесь, что у вас есть два класса с неудобными интерфейсами:
    - *полезный сервис* — служебный класс, который вы не можете изменять (он либо сторонний, либо от него зависит другой
      код);
    - *один или несколько клиентов* — классов приложения, несовместимых с сервисом из-за неудобного или несовпадающего
      интерфейса.
2. Опишите клиентский интерфейс (Target), через который классы приложения смогли бы использовать сторонний класс.
3. Создайте класс адаптера (Adapter), реализовав этот интерфейс.
4. Поместите в адаптер поле-ссылку на объект-сервис (Adaptee). В большинстве случаев, это поле заполняется объектом,
   переданным в конструктор адаптера. В случае простой адаптации этот объект можно передавать через параметры методов
   адаптера.
5. Реализуйте все методы клиентского интерфейса в адаптере. Адаптер должен делегировать основную работу сервису.
6. Приложение должно использовать адаптер только через клиентский интерфейс. Это позволит легко изменять и добавлять
   адаптеры в будущем.

#### Вариации

Адаптер можно реализовать двумя способами:

- **Адаптер объектов** (Object Adapter) — использует композицию: хранит ссылку на адаптируемый объект.
- **Адаптер классов** (Class Adapter) — использует наследование: наследуется от адаптируемого класса.

Оба подхода описаны в предлагаемых книгах.

В данном учебнике мы рассматриваем подход, основанный на **композиции** (адаптер объектов). Он предпочтительней, так как:

- Наследование создаёт жёсткую связь с классом-родителем.
- Изменение класса-родителя может непредсказуемо повлиять на адаптер.
- Класс не всегда можно унаследовать (например, если он `final`).
- Композиция позволяет добиться большей гибкости и переиспользуемости.

### Примеры паттерна адаптер в стандартной библиотеке Java

- Метод `Arrays.asList()` адаптирует массив к интерфейсу `List`. Возвращаемый объект реализует интерфейс List,
  позволяя работать с массивом как с коллекцией.
- Классы `InputStreamReader` и `OutputStreamWriter` адаптируют байтовые потоки (`InputStream`
  и `OutputStream`) к символьным потокам (`Reader` и `Writer`), позволяя работать с байтовыми данными как с
  символами.
- Класс `Scanner` адаптирует поток ввода (`InputStream`, `Readable`) к удобному интерфейсу для чтения данных
  различных типов.

### Примеры

#### [Пример 1](code%2Fexample1_sort%2FSortAdapterMain.java) — Адаптация сортировки массива через List-сервис

Представим ситуацию: нам необходимо отсортировать массив целых чисел, но реализация «мега быстрого» алгоритма
сортировки у нас есть только для списка (`List<Integer>`). Исходный код этой библиотеки мы изменить не можем — класс
объявлен как `final`.

Адаптер преобразует вызов `sort(int[])` в цепочку: `int[]` → `List<Integer>` → сортировка → `List<Integer>` → `int[]`.

**Участники:**
- `Sorter` — целевой интерфейс (сортировка массива).
- `SorterExternalProduct` — внешний сервис (сортировка списка).
- `SorterAdapter` — адаптер, преобразующий массив ↔ список.
- `SortAdapterMain` — демонстрация работы.

**Чему учит пример:** классическая адаптация типов данных. Адаптер преобразует `int[]` в `List<Integer>` и обратно,
чтобы использовать несовместимый API.

#### [Пример 2](code%2Fexample2_logger%2FApplication.java) — Адаптация системы логирования

Наша система использует интерфейс `Logger` с отдельными методами `logInfo()` и `logError()`. Мы хотим подключить
новую библиотеку логирования `NewLogger`, которая имеет единый метод `log(severity, message)`. Вместо переписывания
всего кода мы создаём адаптер.

**Участники:**
- `Logger` — целевой интерфейс (отдельные методы для каждого уровня).
- `NewLogger` — новый логгер с другим интерфейсом (единый метод с параметром severity).
- `LoggerAdapter` — адаптер, делегирующий вызовы новому логгеру.
- `Application` — клиентский код с демонстрацией.

**Чему учит пример:** адаптация сигнатур методов. Несколько специализированных методов адаптируются к одному
универсальному, и наоборот.

#### [Пример 3](code%2Fexample3_notification%2FNotificationMain.java) — Адаптация Telegram Bot API к системе уведомлений

Наша система отправляет уведомления через интерфейс `NotificationService`, который принимает имя пользователя,
заголовок и текст. Мы хотим отправлять уведомления через Telegram, но его Bot API принимает числовой `chatId` и одну
строку текста — совершенно другой формат.

Адаптер решает две задачи: преобразует имя пользователя в chatId (через справочник) и объединяет заголовок с текстом
в единую строку.

**Участники:**
- `NotificationService` — целевой интерфейс (отправка по имени, с заголовком и текстом).
- `TelegramBotApi` — внешний сервис Telegram (отправка по chatId, одной строкой).
- `TelegramNotificationAdapter` — адаптер с преобразованием данных.
- `NotificationMain` — демонстрация работы.

**Чему учит пример:** нетривиальное преобразование данных. Адаптер не просто делегирует вызов, а преобразует типы
параметров (`String` → `long`), объединяет поля и обрабатывает ошибки (пользователь не найден).

### Сравнение примеров

| Пример | Что адаптируется | Тип преобразования |
|--------|------------------|--------------------|
| Сортировка (пример 1) | `int[]` ↔ `List<Integer>` | Преобразование типов данных |
| Логгер (пример 2) | Несколько методов ↔ один метод | Преобразование сигнатур методов |
| Уведомления (пример 3) | Имя → chatId, заголовок+текст → строка | Комплексное преобразование данных |

### Плюсы данного паттерна

- **Повторное использование кода:** Позволяет использовать существующие классы с несовместимыми интерфейсами без
  изменения их кода.
- **Снижение связанности:** Клиентский код зависит только от интерфейса (Target), а не от конкретных адаптируемых
  классов.
- **Гибкость:** Можно легко адаптировать новые системы или библиотеки без необходимости переписывать существующий код.
- **Упрощение интеграции:** Облегчает подключение сторонних библиотек или компонентов в проект.
- **Принцип открытости/закрытости (OCP):** Можно добавлять новые адаптеры, не изменяя существующий клиентский код.

### Недостатки данного паттерна

- **Усложнение структуры:** Добавляет дополнительный слой абстракции, что увеличивает количество классов в системе.
- **Избыточность:** В некоторых случаях может быть излишним, особенно если интерфейсы можно изменить напрямую.
- **Производительность:** Дополнительное преобразование данных (как в примере с сортировкой: `int[]` → `List` → `int[]`)
  создаёт накладные расходы.
- **Надёжность:** Требует тщательной реализации адаптера, чтобы избежать ошибок и потерь при преобразовании данных.

### Заключение

Паттерн проектирования Адаптер является мощным инструментом для интеграции различных компонентов и систем с
несовместимыми интерфейсами. Он позволяет сохранять существующий код неизменным, обеспечивая при этом гибкость и
расширяемость приложения. В стандартной Java библиотеке паттерн Адаптер широко используется для адаптации различных
интерфейсов и классов, обеспечивая удобство и типовую безопасность при работе с коллекциями, потоками ввода-вывода и
другими компонентами.

При правильном применении паттерн Адаптер способствует созданию более модульного, гибкого и поддерживаемого кода,
облегчая интеграцию новых технологий и систем без необходимости радикальных изменений в существующей архитектуре.
