# Правила именования

## Именование аббревиатур в именах классов, полей и методов

Любые аббревиатуры любой длины (2 и более символов) следует писать CamelCase'ом.

**Пример**

```java
// Неправильно
NTLMLDAPHTTPAuth
HTMLColumn
JSONView

// Правильно
NtlmLdapHttpAuth
HtmlColumn
JsonView
```

## Именование констант

Константы всегда должны быть написаны в верхнем регистре. Между словами должен стоять символ подчеркивания.

**Пример**

```java
// Неправильно
protected static final String FIRSTNAME = "firstname";
protected static final String LASTNAME = "lastname";

// Правильно
protected static final String FIRST_NAME = "firstname";
protected static final String LAST_NAME = "lastname";
```

**Пример**

```java
// Неправильно
public static final int INDEX_OF_NEXT_RECURSIVE_PARAM = 1;
public static final int INDEX_BEGIN_PARAM_WITHOUT_RECURSIVE = 1;

// Правильно
public static final int NEXT_RECURSIVE_PARAM_INDEX = 1;
public static final int BEGIN_PARAM_WITHOUT_RECURSIVE_INDEX = 1;
```

> **Зачем это делать**
>
> В Java существует [соглашение](https://www.oracle.com/java/technologies/javase/codeconventions-namingconventions.html)
> об именовании констант. Несмотря на то, что оно необязательное, соблюдение этого правила помогает отличать константы
> от обычных переменных, повысить понимание значения константы и читаемости кода, а также привести его к более
> единообразному стилю

## Именование в enum

Все элементы enum должны именоваться в **UPPER_SNAKE_CASE** (заглавные буквы, слова разделены подчеркиванием).

**Пример enum**

```java
enum OrderStatus {
    AWAITING_PAYMENT,
    PROCESSING,
    SHIPPED,
    DELIVERED,
    CANCELLED_BY_USER
}
```

## Названия переменных и параметров должны отображать их смысловую нагрузку

По названию переменной разработчик должен сразу понимать, какие данные в ней хранятся. Это правило относится ко всем
типам переменных, далее рассмотрены лишь некоторые примеры.

Например, если внутри параметра/переменной находится идентификатор, то в имени также должно находиться указание на Id.:

**Неправильно**

```java
List<Student> getStudentsNeedEducation(String eduPlan, String facultyId,
                                       boolean filterTypes, List<String> types,
                                       boolean filterPlaces, List<String> places,
                                       boolean filterProviders, List<String> providers,
                                       boolean filterDirections, List<String> directions);
```

Как по этим названиям понять, что находится внутри directions? Там названия, коды, или идентификаторы?

**Правильно**

```java
List<Student> getStudentsNeedEducation(String eduPlanId, String facultyId,
                                       boolean filterTypes, List<String> typeIdList,
                                       boolean filterPlaces, List<String> placeIdList,
                                       boolean filterProviders, List<String> providerIdList,
                                       boolean filterDirections, List<String> directionIdList);
```

Если переменная является флагом (relocation) или именем (vacancy), то в названии должно быть указание на это.

**Неправильно**

```java
String vacancy = getVacancyName();

Sql getVacanciesWithCandidates(String relocation)
```

Непонятно, какие данные по вакансии находятся в переменной, и что означает relocation: имя, идентификатор?

**Правильно**

```java
String vacancyName = getVacancyName();

Sql getVacanciesWithCandidates(String relocationYesFlag)
```

### Именование Map

При именовании Map необходимо называть переменную в формате "keyToValueMap" с минимальным описанием key и value, чтобы
было понятно, за что отвечает каждый член мапы.

**Неправильно**

```java
Map<String, StudentDates> datesMap = mapStudentDates(studentList);
```

Непонятно, что является ключом в этой мапе - нужно идти смотреть внутри кода, как она заполняется.

**Правильно**

```java
Map<String, StudentDates> studentIdToStudentDatesMap = mapStudentDates(studentList);
```

## Именование классов при наследовании

При наследовании от класса/реализации интерфейса имя дочернего класса (включая внутренние приватные классы) должно
сохранять суффикс предка.

**Пример**

```java
// Неправильно
public enum FilterCategory implements ButtonAction {
    // Some code
}

// Правильно
public enum FilterCategoryAction implements ButtonAction {
    // Some code
}
```

> **Зачем это делать**
>
> При именовании потомка с суффиксом предка увеличивается скорость понимания типа класса, его возможных методов. При
> таком подходе по названию класса можно найти нужный тип класса. Например, нам необходимо добавить логику в экран
> студента, в поиске по названию класса мы вводим *Student*. Так как все классы имеют суффикс родителя, то мы
> сразу можем понять, что нам нужно изменять класс StudentFrame, а не StudentAction.

## Именование переменных, имеющих обобщенный тип

При использовании названия обобщенного типа в названии переменной стоит указывать его в качестве суффикса.

**Пример**

```java
// Неправильно
List<PreviewConditions> listOfConditions = new ArrayList<>();

// Правильно
List<PreviewConditions> conditionList = new ArrayList<>();
```

Для Optional нет строгого правила указания суффикса в конце. Часто это помогает разграничить переменные по смыслу. Если
решили указать суффикс, то он должен быть понятным (например, Optional или Opt, а не O).

# Организация java-файла

## Порядок расположения методов в абстрактном классе

Абстрактные методы всегда следует располагать сначала. Это нужно для того, чтобы сразу видеть все методы, которые нужно
реализовать у наследника.

**Неправильно**

```java
public abstract class AbstractOrderProcessor {

    public abstract int getOrderType();

    public abstract String getOrderDescription(Order order);

    public OrderException getException(Order order, OrderService orderService) {
        return orderService.createException(order, getOrderType(), getExceptionMessage());
    }

    public abstract String getExceptionMessage();

    public void checkOrder(Order order, OrderService orderService) {
        if (order.isEmpty()) {
            throw getException(order, orderService);
        }
    }
}
```

В данном случае абстрактный метод getExceptionMessage() должен быть выше, чем getException().

**Правильно**

```java
public abstract class AbstractOrderProcessor {

    public abstract int getOrderType();

    public abstract String getOrderDescription(Order order);

    public abstract String getExceptionMessage();

    public OrderException getException(Order order, OrderService orderService) {
        return orderService.createException(order, getOrderType(), getExceptionMessage());
    }

    public void checkOrder(Order order, OrderService orderService) {
        if (order.isEmpty()) {
            throw getException(order, orderService);
        }
    }
}
```

Статические методы всегда находятся в конце класса после всех остальных.

## Соблюдение порядка написания параметров

При написании методов, связанных общей логикой, нужно придерживаться единого порядка параметров. Это в особенности
касается методов, содержащих несколько параметров одного типа, т.к. с легкостью можно допустить ошибку, передав значения
не в правильном порядке.

Например, в следующем блоке кода, в каких-то случаях String channel - первый параметр, а ожидаемый ответ - второй, в
каких-то - наоборот. Так делать не стоит.

**Неправильно**

```java
void assertRpcErrorResponse(String channel, String message);

void assertChannelJsonMessage(String expectedMessageBody,
                              String channel,
                              Map<String, String> expectedHeaders,
                              int timeoutSeconds);

Map<String, String> assertChannelJsonMessage(String expectedMessageBody, String channel);
```

**Правильно**

```java
void assertRpcErrorResponse(String channel, String message);

void assertChannelJsonMessage(String channel,
                              String expectedMessageBody,
                              Map<String, String> expectedHeaders,
                              int timeoutSeconds);

Map<String, String> assertChannelJsonMessage(String channel, String expectedMessageBody);
```

Также хорошей практикой будет соблюдение соответствия порядка полей класса и параметров конструктора.

**Неправильно**

```java
String roomId;
List<Role> roles;

public Room(List<Role> roleList, String roomId) {
    this(roomId, roleList.stream().map(Role::new).collect(Collectors.toList()));
}
```

**Правильно**

```java
String roomId;
List<Role> roles;

public Room(String roomId, List<Role> roleList) {
    this(roomId, roleList.stream().map(Role::new).collect(Collectors.toList()));
}
```

> **Зачем это делать**
>
> Нужно стремиться к согласованности и порядку внутри класса, чтобы в нем было проще ориентироваться и пользоваться им.

## Комментарии

Комментарии должны объяснять **почему** код работает именно так, а не **что** он делает. Комментарий всегда должен быть
понятен всем, кто его читает.

### Примеры неправильных комментариев

**Очевидный комментарий, дублирующий код:**

```java
// увеличиваем i на 1
i++;

// конструктор
public Student(String name) {
    this.name = name;
}
```

**Бессмысленный комментарий:**

```java
// метод
public void process() { ... }
```

### Примеры правильных комментариев

**Объясняет причину выбора структуры данных:**

```java
// Используем LinkedHashMap, чтобы сохранить порядок вставки элементов
Map<String, Integer> nameToScoreMap = new LinkedHashMap<>();
```

**Объясняет неочевидное поведение:**

```java
// Сортируем по убыванию даты, чтобы последние заказы были первыми в списке
orders.sort(Comparator.comparing(Order::getDate).reversed());
```

**Объясняет бизнес-правило:**

```java
// Скидка применяется только к заказам старше 30 дней (требование заказчика)
if (ChronoUnit.DAYS.between(order.getDate(), LocalDate.now()) > 30) {
    applyDiscount(order);
}
```

> **Зачем это делать**
>
> Нужно содержать в порядке как код, так и комментарии, чтобы такой код было удобно читать и понимать. Также наличие
> мусорных комментариев приводит к тому, что остальные правила написания кода тоже перестают соблюдаться.

## Закомментированный код

При внесении кода в систему контроля версий не должно быть закомментированного кода.
Без комментирования кода можно обойтись, так как:

- Если это старый код, то его можно при необходимости посмотреть в гите.
- Если это новый код и работа над ним завершена, то в таком случае точно не должно оставаться никакого
  закомментированного кода.
- Если работа над кодом не завершена и сейчас нужно приостановить работу над ним и сохранить изменения, то можно
  воспользоваться функцией Shelve / Stash.

**Неправильно**

```java
public class OrderService {

    public double calculateTotal(Order order) {
        double total = order.getSubtotal();
        // double tax = total * 0.13;
        // total += tax;
        // if (order.hasDiscount()) {
        //     total -= order.getDiscount();
        // }
        return total;
    }
}
```

**Правильно**

```java
public class OrderService {

    public double calculateTotal(Order order) {
        double total = order.getSubtotal();
        return total;
    }
}
```

> **Зачем это делать**
>
> 1. Бережем время и усилия;
> 2. К такому коду возникает много вопросов - почему он закомментирован, зачем он нужен, что с ним делать и т.д.;
> 3. Если такой код разрастется по разным веткам, то от него становится сложно окончательно избавиться - на это нужно
     тратить время, которое могло бы пойти на разработку новых фич.

# Обработка исключений

В коде могут возникать исключения и при необходимости их нужно отлавливать и обрабатывать.

### Все исключения должны быть обработаны внутри программы

Нельзя пробрасывать исключения из метода `main`. Все исключения должны быть обработаны внутри программы.

**Неправильно**

```java
public class Main {

    public static void main(String[] args) throws Exception {
        List<String> lines = Files.readAllLines(Path.of("data.txt"));
        System.out.println(lines);
    }
}
```

**Правильно**

```java
public class Main {

    public static void main(String[] args) {
        try {
            List<String> lines = Files.readAllLines(Path.of("data.txt"));
            System.out.println(lines);
        } catch (IOException e) {
            System.err.println("Ошибка при чтении файла: " + e.getMessage());
        }
    }
}
```

### Обрабатывайте конкретные типы исключений

В блоке catch лучше использовать более конкретные типы исключений, а не Exception. Может пойматься какое-то
исключение, которое вы не ждали, и для него может потребоваться какая-то специфическая обработка.

**Неправильно**

```java
try {
    int number = Integer.parseInt(input);
    double result = 100.0 / number;
} catch (Exception e) {
    System.err.println("Ошибка: " + e.getMessage());
}
```

**Правильно**

```java
try {
    int number = Integer.parseInt(input);
    double result = 100.0 / number;
} catch (NumberFormatException e) {
    System.err.println("Некорректный формат числа: " + e.getMessage());
} catch (ArithmeticException e) {
    System.err.println("Деление на ноль: " + e.getMessage());
}
```

Также не нужно ловить Throwable, так как таким образом можно поймать наследников Error. Из документации: "An Error is
a subclass of Throwable that indicates serious problems that a reasonable application should not try to catch."

### Отлавливание исключений

- При отлавливании исключения оно должно быть залогировано, или же, если его нужно обработать
  позже, оно может быть проброшено дальше. Но должно происходить что-то одно - или логирование, или пробрасывание. Два
  раза одна и та же информация в лог попасть не должна.
- Не нужно игнорировать исключение, иначе информация о нем уже не сможет быть восстановлена.
- В finally блоке никогда не должно быть исключений. Если вы решили пробросить исключение выше и при выполнении finally
  блока возникло исключение, то в результате выброшено будет оно, а информация об изначальном исключении просто
  пропадет.

### Почему нельзя игнорировать исключения

- При игнорировании исключений через пустой блок catch мы теряем информацию о возможной ошибке
- Даже если исключение является ожидаемым и не является ошибкой, его логирование позволяет получить информацию о том,
  как часто оно возникает и при каких входных данных, тем самым упрощая последующую отладку
- Игнорирование ожидаемых ошибок может привести к тому, что будут проигнорированы другие непредвиденные ошибки этого же
  типа

**Неправильно**

```java
public Order buildOrder(String productName) {
    Order order = new Order();
    try {
        Product product = productService.findByName(productName);
        order.setProduct(product);
    } catch (IllegalArgumentException e) {
        // Не для всех продуктов можно создать заказ, так что игнорируем
    }
    return order;
}
```

Даже если исключение не является ошибкой, информация о нем все равно должна быть залогирована:

**Правильно**

```java
public Order buildOrder(String productName) {
    Order order = new Order();
    try {
        Product product = productService.findByName(productName);
        order.setProduct(product);
    } catch (IllegalArgumentException e) {
        log.warn("Не удалось найти продукт с именем " + productName, e);
    }
    return order;
}
```

# Объявление и инициализация полей и переменных

### Выбор места инициализации поля

Если поле объявляется в классе, то оно обязательно должно быть явно инициализировано. Нельзя инициализировать такие поля
через метод, вызов которого не гарантирован.
Поля класса инициализируем в конструкторе, если:


1. Сложная логика получения значения
2. Необходимо передать данные "извне"

В остальных случаях всегда инициализируем переменные в местах их декларации.

Как не нужно делать:

**Неправильно**

```java
private final Color seriesColor;
private final Map<Integer, Color> categoryColors;

public ChartSeries(int nameRow, int categoryColumn, ChartCoordinates coordinates) {
    this(nameRow, categoryColumn, coordinates, Color.getEmpty());
}

public ChartSeries(int nameRow, int categoryColumn, ChartCoordinates coordinates, Color seriesColor) {
    super(nameRow, categoryColumn, coordinates);
    categoryColors = new HashMap<>(); // Неправильно
    this.seriesColor = seriesColor;
}
```

Как нужно делать:

**Правильно**

```java
private final Color seriesColor;
private final Map<Integer, Color> categoryColors = new HashMap<>(); // Правильно

public ChartSeries(int nameRow, int categoryColumn, ChartCoordinates coordinates) {
    this(nameRow, categoryColumn, coordinates, Color.getEmpty());
}

public ChartSeries(int nameRow, int categoryColumn, ChartCoordinates coordinates, Color seriesColor) {
    super(nameRow, categoryColumn, coordinates);
    this.seriesColor = seriesColor;
}
```

### Константы нельзя инициализировать через методы

Константы нельзя инициализировать через методы. Недостатки инициализации через метод:

1. При изменении методов не всегда смотрят все места использования и могут изменить значение константы.
2. В методе может быть код не готовый к использованию на момент статической реализации.
3. В тестах нельзя использовать статик методы, нужно будет подменять результат.

### Нельзя объявлять переменную без задания значения

Если в коде присутствует объявление локальной переменной, но отсутствует код ее инициализации, это сигнализирует о
плохом коде. Переменная будет инициализирована где-то далее в коде, скорее всего, внутри if/else или switch. Это
усложняет понимание такого кода.

Пример 1.

**Неправильно**

```java
Student student;
if (StringHelper.isNotEmpty(studentId)) {
    student = new Student(studentId);
} else if (StringHelper.isNotEmpty(groupId)) {
    student = new Student(groupId);
} else {
    student = new Student();
}
```

**Правильно (Вариант 1: инициализировать переменную однажды, а затем настроить)**

```java
Student student = new Student();
if (StringHelper.isNotEmpty(studentId)) {
    student.setId(studentId);
} else if (StringHelper.isNotEmpty(groupId)) {
    student.setId(groupId);
}
```

**Правильно (Вариант 2: вынести создание значения в отдельный метод с множественными return)**

```java
private Student createStudent(String studentId, String groupId) {
    if (StringHelper.isNotEmpty(studentId)) {
        return new Student(studentId);
    } else if (StringHelper.isNotEmpty(groupId)) {
        return new Student(groupId);
    } else {
        return new Student();
    }
}
```

Пример 2.

**Неправильно**

```java
double cost;
if (isNewOrder()) {
    cost = newStageCost;
} else {
    cost = newStageCost - getOldCost();
}
```

Нужно использовать тернарный оператор. Или если так сделать нельзя, то нужно выносить в отдельный
метод с несколькими return. Например этот код можно переделать вот так:

**Правильно**

```java
double cost = isNewOrder() ? newStageCost : newStageCost - getOldCost();
```

> **Зачем это делать**
>
> Такая инициализация ухудшает читаемость кода, что усложняет поддержку кода и приводит к лишним затратам времени. Кроме
> того, переменная без значения может привести к NPE, что опять же вынудит затрачивать лишнее время на поддержку.

### Нельзя переприсваивать значение переменной

Переприсваивание ухудшает читаемость кода, может привести к случайному перетиранию данных (если передается в качестве
параметра).

**Неправильно**

```java
@Override
protected Result doRun(Context context) {
    Product product = productService.findById(id);
    Content content = Content.createDefault();
    content.setName(product.getName());
    content = contentService.save(content).getUpdatedBean();
    product.setContentId(content.getId());
    productService.save(product);
    return new Result();
}
```

**Правильно**

```java
@Override
protected Result doRun(Context context) {
    Product product = productService.findById(id);
    Content content = Content.createDefault();
    content.setName(product.getName());
    Content updatedContent = contentService.save(content).getUpdatedBean();
    product.setContentId(updatedContent.getId());
    productService.save(product);
    return new Result();
}
```

### Объявление параметризованных классов

При объявлении переменной/поля параметризованного класса необходимо указывать в угловых скобках все классы параметров.
Иначе разработчику сложнее понять какой тип параметра содержит в себе параметризованный класс.

> **Зачем это делать**
>
> 1. Параметризация повышает читаемость кода - сразу понятно, экземпляры какого класса хранятся в переменной.
> 2. Благодаря параметризации все операции приведения типов выполняются автоматически и неявно.
> 3. Параметризация обеспечивает типовую безопасность типов. Если параметризованный тип не указать, то по умолчанию
     будет подставлен Object, так как в Java все классы наследуются от него. Это может привести к ошибкам, связанным с
     операциями над объектами.

Пример 1

**Неправильно**

```java
// Непонятно какие цены в себе содержит ArrayList. int? double?
ArrayList prices = new ArrayList();
...
// При получении числа из коллекции необходимо дополнительно приводить к нужному классу
int firstPrice = (Integer) prices.get(0);
```

**Правильно**

```java
ArrayList<Integer> prices = new ArrayList<>();
...
int firstPrice = prices.get(0);
```

Пример 2

**Неправильно**

```java
@Autowired
private FileService fileService;
...
File file = fileService.getFile(cover);
```

Если у параметризованного класса используется метод не зависящий от типа параметра или тип нет возможности понять,
допускается использование `<?>`.

**Правильно**

```java
@Autowired
private FileService<?> fileService;
...
File file = fileService.getFile(cover);
```

### В каких случаях нужно объявить поле как final

- Поле инициализируется в конструкторе.
- Поле инициализируется сразу при объявлении.
- Поле является иммутабельной коллекцией.

С помощью final отмечаются поля, которые инициализируются только один раз. Отмечать такое поле как final технически
необязательно, но если этого не сделать, то:

- останется возможность ошибки: разработчик опечатается и переприсвоит значение;
- при чтении кода возникнут вопросы: "Где это поле изменяется? И как это повлияет на остальной код?". Модификатор final
  говорит разработчику, что лишние сценарии рассматривать не требуется.

Следовательно, при использовании final для полей, код будет проще восприниматься, т.к. значение такого поля будет
неизменно.

### Модификатор final защищает ссылку, а не объект

При этом в рассмотренных случаях после инициализации полей нельзя в дальнейшем при вызове каких-то методов эти
поля модифицировать. Модификатор `final` защищает только ссылку на объект, но не сам объект. Это значит, что вызов
методов, изменяющих внутреннее состояние объекта (например, `set()`, `add()`, `put()`), технически возможен, но
нарушает семантику `final` - разработчик ожидает, что `final`-поле не меняется.

**Неправильно**

```java
private final AtomicReference<Integer> state = new AtomicReference<>();

int getState() {
    state.set(new Random().nextInt()); // мутация через метод - нарушает семантику final
    return state.get();
}
```

**Правильно**

```java
private final int state;

public MyClass() {
    this.state = new Random().nextInt(); // значение задается один раз при создании
}

int getState() {
    return state;
}
```

### Нельзя хранить временные данные в полях синглтона

При многопоточной обработке может быть попытка одновременного доступа и модификации поля в классе, имеющем один
экземпляр для всей системы.

**Что может пойти не так:** Под одним пользователем выполняется метод, который кладет в поле какие-то данные,
обрабатывает и возвращает. В это же время под другим пользователем в это поле добавляются другие данные во время
обработки первым. Это приведет к тому, что первому пользователю вернутся данные, измененные вторым пользователем.

Такую ошибку сложно поймать на тестах, т.к. тестирует обычно один человек в одном потоке - возможно отловить только при
нагрузочном тестировании. Такую ошибку сложно локализовать, т.к. в логах может не быть вывода о том, как используется
переменная из синглтона.

**Неправильно**

```java
public class OrderService {

    private Map<String, Boolean> orderIdToDeliveryMap = new HashMap<>();

    private void updateOrderDelivery(List<Order> orders) {
        ...
        orderIdToDeliveryMap.put(orderId, hasDelivery);
    }
}
```

При необходимости хранения информации между вызовами метода, стоит использовать вспомогательный класс или
добавить аргумент к вызываемому несколько раз методу, и передать туда объект, созданный специально для этих вызовов.

**Правильно**

```java
public class OrderService {

    public void updateDeliveryModel(OrderModel orderModel) {
        Map<String, Boolean> orderIdToDeliveryMap = new HashMap<>();
        Iterator<Order> iterator = getIterator();
        iterator.forEach(orders -> {
            updateOrderDelivery(orders, orderIdToDeliveryMap);
        });
    }

    private void updateOrderDelivery(List<Order> orders, Map<String, Boolean> orderIdToDeliveryMap) {
        ...
        orderIdToDeliveryMap.put(orderId, hasDelivery);
    }
}
```

# Параметры методов

## Boolean в параметрах метода

Не нужно создавать метод с *boolean* параметрами (только если это не сеттер). Это правило актуально также для
конструкторов.

Для решения проблемы можно использовать наследование (*@Override)*, декомпозицию и др.

**Неправильно**

```java
private Component createReadOnlyLink() {
    return createLink(true);
}

private Component createWriteableLink() {
    return createLink(false);
}

private Component createLink(boolean isReadOnly) {
    Component linkContainer = new Component("empty");
    String link = "http://localhost";
    linkContainer.addChild(Link.create(link));
    linkContainer.setName("url");
    linkContainer.setReadOnly(isReadOnly);
    return linkContainer;
}
```

**Правильно**

```java
private Component createReadOnlyLink() {
    Component linkComponent = createLink();
    linkComponent.setReadOnly(true);
    return linkComponent;
}

private Component createWriteableLink() {
    Component linkComponent = createLink();
    linkComponent.setReadOnly(false);
    return linkComponent;
}

private Component createLink() {
    Component linkContainer = new Component("change");
    String link = "http://localhost";
    linkContainer.addChild(Link.create(link));
    linkContainer.setName("url");
    return linkContainer;
}
```

> **Зачем это делать**
>
> Главная причина в том, что boolean-параметры усложняют чтение кода и снижают его поддерживаемость.
>
> - Чтение: зачастую по вызову метода с boolean-параметром (если это не простой сеттер) непонятно, на что этот параметр
    влияет внутри метода - например, он переключает какую-то логику, но какую? Нужно идти внутрь и смотреть - это
    неэффективно;
> - Поддерживаемость: например, есть логика, у которой раньше было 2 варианта поведения А и Б, а стало 3 - А, Б и В, или
    5. Как добавить эти новые варианты в метод, который раньше работал через boolean-параметр (или А, или Б)?
    Перед разработчиком возникает проблема на ровном месте, над которой надо думать и тратить время.

## Передача null в параметрах метода

*Проблема*

Если в каких-то случаях вы передаете null в параметрах метода, то нужно проверять на этот null внутри метода. Но
разрабатывая содержимое метода нельзя точно сказать, как его будут использовать, поэтому теоретически нужно проверять
все параметры на null. Если мы так будем делать, то будет огромное количество ненужного кода.

*Вывод*

Поэтому существует правило, что нельзя никогда передавать null внутрь метода и нужно строить свой код исходя из этого
принципа.

**Пример**

```java
// Неправильно
sendEmail(sender, receiver, "", null, Collections.emptyList());

// Правильно
sendExternalEmail(sender, receiver);
```

## Optional в параметрах методов

По аналогии с передачей null, при передаче Optional всегда нужно проверять его на null внутри метода, отсюда возможный
ненужный код. Изначально Optional создавали только для возвращения значений, поэтому какая-то его обработка, передача,
использование в полях класса - нежелательно.

Также рассмотрим ситуацию: есть интерфейс сервиса, в нем n-ое количество методов, каждый из которых содержит *Optional*
в параметре. Правило из этого пункта вынуждает создавать взамен n*2 методов (с не null параметром и без). Так делать не
нужно. Можно вынести этот параметр в поле сервиса или использовать фабрику сервисов.

Например, был интерфейс с методами которые опционально принимают *userId*:

**Старый интерфейс**

```java
public interface RecordPlayer {

    void prepareRecording(String roomId, UUID recordId, UUID translationRecordId, Optional<String> userId);

    void prepareScreenRecording(String roomId, UUID recordId, UUID translationRecordId, Optional<String> userId);
}
```

В интерфейсе оставили методы не использующие *userId*:

**Новый интерфейс**

```java
public interface RecordPlayer {

    void prepareRecording(String roomId, UUID recordId, UUID translationRecordId);

    void prepareScreenRecording(String roomId, UUID recordId, UUID translationRecordId);
}
```

И создали реализацию, которая содержит поле *userId*:

**Новая реализация**

```java
public class RecordPlayerWithUserId implements RecordPlayer {

    private final RecordRequestService requestService;
    private final RoomStore roomStore;
    private final String userId;

    public RecordPlayerWithUserId(RecordRequestService requestService,
                                  RoomStore roomStore,
                                  String userId) {
        this.requestService = requestService;
        this.roomStore = roomStore;
        this.userId = userId;
    }

    @Override
    public void prepareRecording(String roomId, UUID recordId, UUID translationRecordId) {
        prepareRecording(roomId, getRoomIds(recordId).getConferenceRoomId(), translationRecordId, userId);
    }

    @Override
    public void prepareScreenRecording(String roomId, UUID recordId, UUID translationRecordId) {
        prepareRecording(roomId, getRoomIds(recordId).getSharingRoomId(), translationRecordId, userId);
    }

    private RoomIds getRoomIds(UUID recordId) {
        return roomStore.getRoomIds("record" + recordId);
    }

    private void prepareRecording(String roomId, String targetRoomId, UUID translationRecordId, String userId) {
        requestService.sendAndExpect(roomId, new PreparePlayoutRequestBody(targetRoomId, translationRecordId, userId));
    }
}
```

## Вызов метода с несколькими параметрами в параметрах метода

Если в параметрах метода вызывается еще один метод с несколькими параметрами или вызывается метод, в параметрах которого
тоже вызывается метод, то нужно вынести получение значения этого параметра в переменную или в отдельный метод.

**Неправильно**

```java
if (manager.isEnabled()) {
    sender.sendNotification(
        manager,
        studentService.list(
            Student.class,
            Student.ID,
            getNameIdSet(getValueSet(studentList, Student.GROUP), Group.TEACHER_ID)
        )
    );
}
```

**Правильно**

```java
Set<Group> groups = getValueSet(studentList, Student.GROUP);
Set<String> idSet = getNameIdSet(groups, Group.TEACHER_ID);
List<Student> receiversList = studentService.list(Student.class, Student.ID, idSet);
if (manager.isEnabled()) {
    sender.sendNotification(manager, receiversList);
}
```

> **Зачем это делать**
>
> Это значительно упрощает читаемость кода: не требуется разбираться в цепочке методов, чтобы понять, какой метод что
> вернет. В случае с примером выше из названия переменной "receiversList" становится понятно, что она должна возвращать,
> и нам даже не потребовалось вникать в саму цепочку методов, которые приводят к результату.

# Результат метода

## Возвращение результата метода

Необходимо придерживаться раннего возврата из метода (использовать множество return).

**Неправильно**

```java
public String getSystemLookAndFeelClassName() {
    String result = getCrossPlatformLookAndFeelClassName();
    String systemLAF = System.getProperty("swing.systemlaf");
    if (systemLAF != null) {
        result = systemLAF;
    }
    String osName = System.getProperty("os.name");
    if (osName != null) {
        if (osName.contains("Windows")) {
            result = "com.sun.java.swing.plaf.windows.WindowsLookAndFeel";
        } else {
            String desktop = System.getProperty("sun.desktop");
            if ("gnome".equals(desktop)) {
                result = "com.sun.java.swing.plaf.gtk.GTKLookAndFeel";
            }
            if (osName.contains("Solaris") || osName.contains("SunOS")) {
                result = "com.sun.java.swing.plaf.motif.MotifLookAndFeel";
            }
        }
    }
    return result;
}
```

**Правильно**

```java
public static String getSystemLookAndFeelClassName() {
    String systemLAF = System.getProperty("swing.systemlaf");
    if (systemLAF != null) {
        return systemLAF;
    }
    String osName = System.getProperty("os.name");
    if (osName != null) {
        if (osName.contains("Windows")) {
            return "com.sun.java.swing.plaf.windows.WindowsLookAndFeel";
        } else {
            String desktop = System.getProperty("sun.desktop");
            if ("gnome".equals(desktop)) {
                return "com.sun.java.swing.plaf.gtk.GTKLookAndFeel";
            }
            if (osName.contains("Solaris") || osName.contains("SunOS")) {
                return "com.sun.java.swing.plaf.motif.MotifLookAndFeel";
            }
        }
    }
    return getCrossPlatformLookAndFeelClassName();
}
```

> **Зачем это делать**
>
> Там, где понятно, что результат уже не может быть изменен его стоит сразу возвращать. Так нам не нужно дальше искать в
> этой функции место, где результат может быть переопределен исходя из других условий. К тому же может выполняться
> какой-то лишний код этой функции, хотя результат уже известен (особенно это плохо, если этот код ресурсоемок).

## Метод не должен возвращать null

Если по логике метод может возвращать null, стоит использовать функционал *Optional*.

**Неправильно**

```java
public String getHeadDepartmentIdByStudent(Student student) {
    String departmentId = student.getDepartmentId();
    if (departmentId == null || departmentId.isEmpty()) {
        return null;
    }
    Optional<Department> optionalDepartment = getDepartmentById(departmentId);
    while (optionalDepartment.isPresent() && !optionalDepartment.get().isHead()) {
        String parentId = optionalDepartment.get().getParentId();
        if (parentId == null || parentId.isEmpty()) {
            return null;
        }
        optionalDepartment = findDepartmentById(parentId);
    }
    return optionalDepartment.map(Department::getId).orElse(null);
}
```

**Правильно**

```java
public Optional<String> getHeadDepartmentIdByStudent(Student student) {
    String departmentId = student.getDepartmentId();
    if (departmentId == null || departmentId.isEmpty()) {
        return Optional.empty();
    }
    return getDepartmentById(departmentId)
        .map(dept -> getClosestHeadParentDepartmentId(dept.getId()));
}
```

Также, если метод возвращает список, то нельзя возвращать null, нужно возвращать пустой список.

**Так делать нельзя**

```java
@Override
public List<TabInfo> getTabInfoList(EntityState state) {
    return null;
}
```

> **Зачем это делать**
>
> 1. Если метод может возвращать null, то вызывающий код обязан проверять на null полученные от метода данные. Иначе
     есть риск получить NPE при работе с этими данными. Это очень сильно загромождает код.
> 2. Если метод вернул *null*, а вызывающий код не проверяя передает его дальше, например кладет его в коллекцию или в
     поле другого объекта, то в дальнейшем нужно будет потратить дополнительное время, чтобы найти место, где этот null
     был добавлен.

## Методы не должны возвращать магические строки или магические числа

**Пример 1 — магические числа**

```java
// Неправильно
public double calculateDiscount(double price) {
    if (price > 1000) {
        return price * 0.15;
    }
    return price * 0.05;
}

// Правильно
private static final double PRICE_THRESHOLD = 1000;
private static final double HIGH_DISCOUNT_RATE = 0.15;
private static final double LOW_DISCOUNT_RATE = 0.05;

public double calculateDiscount(double price) {
    if (price > PRICE_THRESHOLD) {
        return price * HIGH_DISCOUNT_RATE;
    }
    return price * LOW_DISCOUNT_RATE;
}
```

**Пример 2 — магические строки**

```java
// Неправильно
public String getRole(User user) {
    if (user.isAdmin()) {
        return "admin";
    }
    return "user";
}

// Правильно
private static final String ADMIN_ROLE = "admin";
private static final String USER_ROLE = "user";

public String getRole(User user) {
    if (user.isAdmin()) {
        return ADMIN_ROLE;
    }
    return USER_ROLE;
}
```

> **Зачем это делать**
>
> Магические строки и числа могут использоваться в нескольких местах системы. Изменение только в одном месте может
> привести к неправильной работе. Вынесение в константы позволяет менять значение в одном месте, не разыскивая все
> зависимые участки кода.

## Нельзя игнорировать результат, возвращаемый родительским методом

Например:

```java
// Неправильно
@Override
protected Result doRun(Context context) {
    super.doRun(context);
    return new Result();
}

// Правильно
@Override
protected Result doRun(Context context) {
    Result result = super.doRun(context);
    // дополнительная логика
    return result;
}
```

Если метод предка возвращает какое-то значение, то его нельзя игнорировать. Это может привести к неправильным
последствиям. Например, если там создается Result со специальным сообщением для пользователя, а здесь это
сообщение игнорируется.

## Методы, возвращающие объекты, в которых лежат неизменяемые данные, должны возвращать иммутабельные версии этих объектов или копии

Если в классе есть поле, где лежит коллекция с фиксированным набором значений, то метод, который отдает эту коллекцию
наружу, должен отдавать ее в немутабельном виде. Например:

**Неправильно**

```java
public class ConversionService {

    private static final Set<String> ALLOWED_EXTENSIONS = new HashSet<>(Set.of(".png", ".jpeg"));

    public Set<String> getAllowedExtensions() {
        return ALLOWED_EXTENSIONS;
    }
}

public class SomeOtherService {

    public void foo() {
        Set<String> allowedExtensions = conversionService.getAllowedExtensions();
        allowedExtensions.add(".gif"); // Изменяет оригинальную коллекцию!
        bar(allowedExtensions);
    }
}
```

В данном случае, при вызове `SomeOtherService#foo` коллекция ALLOWED_EXTENSIONS будет изменена и весь последующий код,
который будет использовать эту коллекцию, будет получать неправильный результат.

**Правильно**

```java
private static final Set<String> ALLOWED_EXTENSIONS = Set.of(".png", ".jpeg");
```

Хорошим правилом считается сразу объявлять коллекции иммутабельными, если не предполагается, что они могут меняться.

> **Зачем это делать**
>
> Основная причина - чтобы другой код не мог изменить эту коллекцию. Ведь в случае изменения могут возникнуть
> непредвиденные ошибки.
>
> Также можно отнести упрощение понимания/читаемости кода: коллекция фиксированная, значит используется в определенных
> ситуациях. Просто так ее менять (добавлением нового значения) не следует.

# Избыточный код

### Не должно быть избыточных переменных

**Пример с избыточными переменными**

```java
// Неправильно
private Student getStudent(String id) {
    Student student = studentService.findById(id);
    return student;
}

// Правильно
private Student getStudent(String id) {
    return studentService.findById(id);
}
```

> **Зачем это делать**
>
> Мы должны стараться писать максимально лаконичный код, потому что чем меньше код класса/метода, тем легче в нем
> ориентироваться.

### Не должно быть дублирования кода

**Пример дублирования кода**

```java
// Неправильно
params.addInteger((recruiterId == null) ? 1 : 0);
params.addInteger((recruiterId != null) ? Integer.valueOf(recruiterId) : -1);
params.addInteger((cityId == null) ? 1 : 0);
params.addInteger((cityId != null) ? Integer.valueOf(cityId) : -1);
params.addInteger((reasonId == null) ? 1 : 0);
params.addInteger((reasonId != null) ? Integer.valueOf(reasonId) : -1);

// Правильно
addNullableParam(params, recruiterId);
addNullableParam(params, cityId);
addNullableParam(params, reasonId);
```

Если вы переписываете метод откуда-то руками или копируете этот метод, то самое время задуматься как можно избавиться от
этого дублирования.

Обычно, если вы хоть раз используете Copy/Paste, это значит, что у вас появилось дублирование и его нужно убирать.

# Преобразования объектов

## Нужно избегать явного приведения типов

**Неправильно**

```java
.findFirst().map(tab -> (ChatTab) tab)
```

Явное приведение типов чревато тем, что если приведение не получится, то ошибка будет на стадии выполнения программы,
что плохо. Поэтому надо стараться избегать этого.

# Иммутабельность

С иммутабельными классами намного проще работать: кода меньше, нет лишних сеттеров, есть гарантия того, что поля данного
объекта не будут изменены где-то еще. Поэтому по возможности поля класса и объекты должны быть иммутабельными.

## Важность создания иммутабельных классов

С иммутабельными классами намного проще работать: кода меньше и он легче читается, нет лишних setter'ов, есть гарантия
того, что поля данного объекта не будут изменены где-то еще. Такой класс всегда инициализируется в одном месте и просто
отдает данные, благодаря чему не нужно искать где они потом могут еще поменяться и почему. Вместо нескольких одинаковых
экземпляров можно использовать один закэшированный объект, что позволяет экономить память. Поэтому по возможности поля
класса и объекты должны быть иммутабельными.

Чтобы класс был иммутабельным, он должен соответствовать следующим требованиям:

- Должен быть объявлен как final, чтобы от него нельзя было наследоваться, иначе дочерние классы могут нарушить
  иммутабельность.
- Все поля класса должны быть приватными в соответствии с принципами инкапсуляции.
- Для корректного создания экземпляра, в нем должны быть параметризованные конструкторы, через которые осуществляется
  первоначальная инициализация полей класса.
- Для исключения возможности изменения состояния после инстанцирования, в классе не должно быть сеттеров.
- Для полей-коллекций необходимо возвращать их глубокие копии (defensive copy), чтобы гарантировать их неизменность.

# Обобщение типов коллекций

## Нельзя обобщать коллекции до Iterable или Collection

При объявлении типа переменной, параметра или возвращаемого значения нельзя обобщать коллекции до `Iterable` или
`Collection`. Максимальный уровень обобщения: `Map`, `Set`, `List`.

### Почему нельзя использовать Iterable

- Нет метода `size()` - невозможно узнать количество элементов без полного обхода
- Нет гарантии повторного обхода - `Iterable` не гарантирует, что по коллекции можно пройти дважды
- Нет удобных методов работы с коллекцией (`contains()`, `isEmpty()`, `toArray()` и т.д.)

### Почему нельзя использовать Collection

- Не различает семантику: множество (уникальные элементы), список (упорядоченные с доступом по индексу), очередь (FIFO/LIFO)
- Вызывающий код не понимает, что именно он получает, и не может безопасно использовать специфичные методы
- При передаче `Collection` теряется информация о контракте: можно ли обращаться по индексу? Есть ли дубликаты?

### Map, Set, List дают четкий контракт

- `List` - упорядоченная коллекция с доступом по индексу, допускает дубликаты
- `Set` - коллекция уникальных элементов, без доступа по индексу
- `Map` - коллекция пар ключ-значение

**Неправильно**

```java
public Iterable<Student> getStudents() {
    return studentRepository.findAll();
}

public void printAll(Collection<Student> students) {
    // Непонятно: это список? множество? Можно ли обратиться по индексу?
    for (Student student : students) {
        System.out.println(student.getName());
    }
}
```

**Правильно**

```java
public List<Student> getStudents() {
    return studentRepository.findAll();
}

public void printAll(List<Student> students) {
    for (Student student : students) {
        System.out.println(student.getName());
    }
}
```
