# Сдача лабораторных работ

**Сдача лабораторных работ происходит только в очном формате.**

1. Необходимо продемонстрировать работу вашей программы на собственной рабочей машине (ноутбук) или на рабочей машине в
   кабинете.
    - Назвать вариант ЛР (можете написать в комменте в Main классе, чтобы не забыть)
    - Дать краткое описание вашей системы: что делает и зачем
    - Запустить программу, "протыкать" несколько базовый сценариев работы
2. Code review
    - Преподаватель просматривает ваш код, попутно задавая вопросы и оставляя замечания
    - При жестких нарушениях реализации паттернов, не соответствии [правилам оформления кода](Стандарты кода.md) -
      работа отправляется на переделку
3. Сдача теории (ответ на вопросы)
4. Демонстрация отчета
    - Отчет демонстрируется на собственной рабочей машине (ноутбуке) или на рабочей машине в аудитории (в таком случае
      лучше приносить PDF-версию)
    - К каждой работе должен быть правильно оформленный отчет с титульным, удовлетворяющий требованиям описанным в
      задании. Без отчета ЛР не считается сданной до конца.
    - Отчет в большей степени формальность, так что сильно его расписывать не стоит, но он должен удовлетворять
      требованиям.

## Замечания:

Для сдачи в аудитории вы можете воспользоваться GitHub-ом, либо принести материалы на флешке.

Сдача осуществляется в перерывах на ЛР, либо пока остальные выполняют задание, либо после завершения ЛР. Можно приходить
сдавать с другими подгруппами в любой день когда преподаватель ведет задания.

> **Для получения автомата необходимо сдать все ЛР до зачетной недели (включительно).**

## Ограничения

### Запрещается

- Использовать операции проверки типов `instanceof` и явное приведение типов. Паттерны как раз предназначены, чтобы
  избегать данных потенциально опасных операций при помощи полиморфизма и тд.
- Использовать паттерн **Итератор** пробрасывая базовый итератор коллекций. Данный паттерн можно использовать только
  если у вас действительно реализуется полноценная логика обхода какой-либо структуры.
- Использовать паттерн **Прототип** просто реализовав его и ни разу не использовав. Однако даже реализацию и
  использование данного паттерна в рамках ЛР оправдать сложно.
- Использовать паттерн **Адаптер**. Поскольку всю ЛР разрабатываете вы и только вы, то в большинстве случаев не должно
  возникать ситуации когда данный паттерн необходим. Если вы по каким-либо причинам создали непреодолимую ситуацию
  которая решается только через адаптер, то у вас явная ошибка в проектировании системы в рамках ЛР. Также не следует
  намеренно создавать такие ситуации, чтобы добавить данный паттерн.

### С осторожностью

- Использовать паттерн **Фасад**. Для оправдания данного паттерна за ним должны быть спрятаны действительно
  большие/сложные вещи, которые трудно достижимы в рамках ЛР. Спрятать в метод последовательный вызов 3ех функций и
  назвать это Фасадом не получится.
