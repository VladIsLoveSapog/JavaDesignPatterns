# wait, notify, notifyAll

## 1. Проблема: зачем нужны эти методы?

Потоки часто должны ждать друг друга. Например, один поток кладёт данные в буфер, а другой их оттуда читает. Если читатель придёт раньше — ему нечего брать; если писатель придёт раньше — некуда класть (буфер занят).

Самое наивное решение — **busy-wait**: поток крутится в цикле и проверяет условие снова и снова. Это работает, но впустую сжигает процессор.

```
while (!dataReady) { /* ничего не делаем, просто ждём */ }
```

Методы `wait()`, `notify()` и `notifyAll()` решают ту же задачу правильно: **поток засыпает и не потребляет CPU**, а другой поток его будит ровно тогда, когда условие выполнено.

---

## 2. Жизненные примеры

| Ситуация | Кто ждёт | Кто будит |
|---|---|---|
| **Сборочная линия** — рабочий кладёт деталь на конвейер, сборщик её забирает | Сборщик ждёт детали; рабочий ждёт, пока сборщик не освободит место | Сборщик будит рабочего; рабочий будит сборщика |
| **Кассир в банке** — клиенты стоят в очереди, кассир обслуживает одного | Клиенты ждут вызова | Кассир вызывает следующего (`notify`) |
| **Загрузка файла** — обработчик не может начать, пока файл не скачан | Обработчик ждёт окончания загрузки | Загрузчик сигнализирует об окончании |
| **Принтер** — несколько программ хотят печатать | Остальные программы ждут | Принтер освобождается и будит очередь |

## 3. Связь с другими механизмами синхронизации

| Механизм | Аналогия | Когда применять |
|---|---|---|
| **mutex** (synchronized) | «Только один входит в комнату» | Защита критической секции от гонок |
| **semaphore** | «N могут входить одновременно» | Ограничение количества параллельных операций |
| **wait / notify** | «Жди сигнала от другого потока» | Координация: один поток ждёт, пока другой что-то сделает |

`wait/notify` всегда работают **внутри** `synchronized` — сначала нужен mutex, потом на нём можно ждать и будить.

---

## 4. Монитор

В Java **каждый объект является монитором**. Монитор — это встроенный замок плюс очередь ожидания. Когда поток входит в `synchronized`-метод, он захватывает монитор объекта. Когда вызывает `wait()` — отпускает монитор и встаёт в очередь ожидания. Когда другой поток вызывает `notify()` — один поток из очереди переводится в состояние "готов к выполнению" и снова борется за монитор.

```
Монитор объекта
├── Замок (lock)       ← захватывается при входе в synchronized
└── Очередь ожидания  ← сюда попадает поток после вызова wait()
```

---

## 5. Правила использования

**1. Только внутри `synchronized`**

`wait()`, `notify()`, `notifyAll()` можно вызывать только в синхронизированном методе или блоке по тому же объекту. Иначе — `IllegalMonitorStateException`.

**2. Всегда `while`, не `if`**

```java
// Неправильно — уязвимо к spurious wakeup
if (!condition) {
    wait();
}

// Правильно — после пробуждения условие проверяется снова
while (!condition) {
    wait();
}
```

*Spurious wakeup* — поток может проснуться без вызова `notify()`. Спецификация JVM это допускает. Цикл `while` защищает от этого.

**3. Обработка `InterruptedException`**

`wait()` выбрасывает `InterruptedException`, если поток был прерван во время ожидания. Стандартная обработка:

```java
try {
    wait();
} catch (InterruptedException e) {
    Thread.currentThread().interrupt(); // восстанавливаем флаг прерывания
}
```

## 6. Методы

### `wait()`

Поток **освобождает монитор** и засыпает до тех пор, пока другой поток не вызовет `notify()` или `notifyAll()` на том же объекте.
После пробуждения поток снова конкурирует за монитор перед продолжением работы.

### `notify()`

Будит **один случайный** поток из очереди ожидания. Остальные продолжают спать.

### `notifyAll()`

Будит **все** ожидающие потоки. Каждый из них после пробуждения конкурирует за монитор; только один получит его первым.

---

### notify() vs notifyAll()

| | `notify()` | `notifyAll()` |
|---|---|---|
| Будит потоков | 1 (случайный) | Все |
| Риск | **Missed notification** — нужный поток не проснётся, если проснулся не тот | Нет (все получат сигнал) |
| Когда подходит | Ровно 1 ожидающий поток, и он точно может продолжить | Несколько ожидающих или разные условия ожидания |

**Missed notification** — ситуация, когда `notify()` разбудил поток A, но нужен был поток B. Поток A видит, что условие для него не выполнено, и снова засыпает. Поток B теперь никто не разбудит — он будет ждать вечно.
`notifyAll()` устраняет эту проблему: все потоки просыпаются и сами проверяют своё условие.

**Практическое правило:** используйте `notifyAll()` по умолчанию; переходите на `notify()` только если уверены, что ожидающий поток ровно один.

## 7. Примеры

### [NotifyMain.java](code/example1_notify/NotifyMain.java)

**Один производитель + один потребитель, `notify()`.**

Производитель кладёт числа 1..5 в однослотовый буфер; потребитель их забирает. При занятом буфере производитель засыпает через `wait()`; при пустом — потребитель. После каждой операции `notify()` будит партнёра.

Показывает: базовый цикл wait/notify, обязательность `while`-условия, освобождение монитора при `wait()`.

---

### [NotifyAllMain.java](code/example2_notify_all/NotifyAllMain.java)

**Один производитель + два потребителя, `notifyAll()` + флаг завершения.**

Производитель кладёт числа 1..5, затем устанавливает флаг `finished` и вызывает `notifyAll()`. Потребители читают данные по очереди; когда буфер пуст и `finished == true`, `consume()` возвращает `null` — это сигнал для выхода.

Показывает: почему `notify()` здесь опасен (missed notification), роль флага завершения, корректное завершение нескольких потоков-потребителей.
